# 角色专业化 (Role Specialization) —— 规划者 (Planner)、审查者 (Critic)、执行者 (Executor)、验证者 (Verifier)

> 2026年最常见的多智能体 (Multi-Agent) 架构拆分模式为：一个智能体负责规划，一个负责执行，另一个负责审查或验证。MetaGPT (arXiv:2308.00352) 将这一模式形式化为嵌入角色提示词 (Role Prompts) 的标准作业程序 (SOPs) —— 涵盖产品经理、架构师、项目经理、工程师与QA工程师 —— 并遵循 `Code = SOP(Team)` 的理念。ChatDev (arXiv:2307.07924) 则通过“对话链 (Chat Chain)”将设计师、程序员、审查员与测试员串联起来，并引入“通信式去幻觉 (Communicative Dehallucination)”机制（即智能体会主动要求补充缺失的细节）。验证者 (Verifier) 是系统的关键支撑：Cemri 等人（MAST, arXiv:2503.13657）的研究表明，多智能体系统的每一次失败均可追溯至验证环节的缺失或失效。普华永道 (PwC) 的报告指出，在 CrewAI 中采用结构化验证循环 (Structured Validation Loops) 可使准确率提升7倍（从10%跃升至70%）。

**类型：** 学习 + 构建
**编程语言：** Python (stdlib)
**前置要求：** 第16阶段 · 04（原始模型 (Primitive Model)），第16阶段 · 05（监督者 (Supervisor)）
**预计耗时：** 约60分钟

## 问题

通用的多智能体系统往往只能产出平庸的输出。就像群聊中的三位程序员，最终只会写出三种风格迥异但同样平庸的代码。即使不断增加智能体数量或交互轮次，依然难以跨越质量门槛。

破局的关键不在于堆砌更多智能体，而在于引入*差异化*的智能体。明确划分角色职责。为审查者配备规划者所不具备的工具。为验证者提供客观的测试套件。此时，系统内部将形成基于事实依据的纠错机制，而非简单的并行猜测。

## 概念

### 四大标准角色

**规划者（Planner）**。读取目标，生成步骤列表或规格说明（spec）。工具：知识检索、文档。输出：结构化计划。

**执行者（Executor）**。逐条读取计划步骤，生成工件（artifact）。工具：实际工作工具（代码编译器、Shell、API 客户端）。输出：工件。

**评审者（Critic）**。对照规划者的意图审查执行者的输出。工具：对工件的只读访问权限、静态分析。输出：附带理由的接受/拒绝决定。

**验证者（Verifier）**。读取工件并运行确定性检查。工具：测试运行器、类型检查器、模式验证器。输出：附带证据的通过/失败结果。

评审者是主观的、带有观点的，通常基于大语言模型（LLM）。验证者是客观的、确定性的，通常基于代码。它们并非同一角色。

### MetaGPT 的标准作业程序（SOP）模式

MetaGPT（arXiv:2308.00352）将软件工程的标准作业程序（SOP）编码为角色提示词：

- **产品经理（Product Manager）** 撰写产品需求文档（PRD）。
- **架构师（Architect）** 产出系统设计。
- **项目经理（Project Manager）** 拆分任务。
- **工程师（Engineer）** 负责实现。
- **QA 工程师（QA Engineer）** 运行测试。

每个角色都有严格的输入/输出模式（schema）。角色提示词明确了该角色*是什么*以及它*必须产出什么*。`Code = SOP(Team)` 这一公式表明——确定性的 SOP 能将大语言模型团队转化为可预测的流水线。

### ChatDev 的通信式去幻觉机制

ChatDev 引入了一项关键机制：当执行者需要计划中未包含的具体细节时，会在继续操作前明确向设计者提问。这有效避免了大语言模型典型的“看似合理地编造细节”的失败模式。

实现方式：角色提示词中包含“当你需要未被提供的具体信息时，在生成输出前按名称询问相关角色。”

### 为什么验证者最为关键

Cemri 等人（MAST）追踪了 1642 次多智能体执行失败案例。其中 21.3% 属于验证缺失——系统交付了未经任何人检查的答案。其余 79% 通常可追溯至“存在静默失败或从未运行的检查”。验证是承担核心负载的角色。

普华永道（PwC）报告（CrewAI 部署案例，2025 年）指出，增加结构化验证循环可将准确率从 10% 提升至 70%。仅凭一个角色就实现了 7 倍的性能提升。

### 评审者与验证者的对比

- 评审者是由大语言模型对工件进行质量审查。具有主观性。容易被看似合理的文本所欺骗。
- 验证者是在工件上运行的确定性程序。具有客观性。提供附带证据的通过/失败判定。

两者应结合使用。评审者能捕捉验证者无法明确表述的风格问题。验证者能发现评审者无法察觉的缺陷，因为这些缺陷仅在运行时才会显现。

### 反模式

系统中的每个角色都是大语言模型，且每个角色的输出都是“我觉得没问题”。这是典型的 MAST 失败模式。请至少添加一个验证者，其通过/失败的判定由代码而非大语言模型决定。

### 框架映射

- **CrewAI** — `Agent(role, goal, backstory)` 是教科书式的角色专业化接口。
- **LangGraph** — 节点可配置专用提示词；边（edges）用于强制约束流水线。
- **AutoGen** — 在群聊（GroupChat）中使用具有单字名称的特定角色 `ConversableAgents`。
- **OpenAI Agents SDK** — 在角色专用智能体（Agents）之间提供交接工具。

## 构建它

`code/main.py` 实现了一个包含 4 个角色（Role）的流水线（Pipeline），用于构建一个简单的 Python 函数：

- **规划器（Planner）** 生成规范（Spec）。
- **执行器（Executor）** 生成代码字符串。
- **审查器（Critic）**（由大语言模型（LLM）模拟）标记明显的问题。
- **验证器（Verifier）** 在沙箱（Sandbox）（使用 `exec`）中针对测试用例运行生成的代码。

演示程序将运行两次：第一次执行器生成正确的代码（审查器与验证器均通过）；第二次执行器生成偏离规范的代码（审查器因代码看似合理而漏掉了缺陷，但验证器因测试失败而成功捕获了该缺陷）。

运行：

python3 code/main.py

## 使用它

`outputs/skill-role-designer.md` 接收一个任务，并生成角色列表（3-5 个角色）、每个角色的输入/输出模式（Schema）以及验证器检查项。在将智能体（Agent）接入框架之前，请先使用此文件。

## 交付它

检查清单：

- **至少包含一个确定性验证器（Deterministic Verifier）。** 绝不能完全依赖大语言模型。
- **为每个角色明确定义输入/输出模式（I/O Schema）。** 规划器返回的是结构化规范，而非自然语言描述；执行器需读取该模式。
- **沟通式去幻觉（Communicative Dehallucination）。** 当信息缺失时，执行器必须向规划器询问，绝不可自行编造。
- **审查器与验证器的执行顺序。** 先运行审查器（成本低，可捕获设计问题），再运行验证器（速度较慢，可捕获代码缺陷）。
- **循环预算（Loop Budget）。** 在交由人工介入前，最多允许 2 轮“审查器-执行器”的修订循环。

## 练习

1. 运行 `code/main.py`，观察验证器如何捕获审查器遗漏的缺陷。添加一项静态分析（Static Analysis）检查（统计 `return` 的出现次数）作为额外的验证器。它能捕获哪些运行时测试（Runtime Test）无法发现的问题？
2. 添加第 5 个角色：“需求分析师（Requirements Analyst）”，负责将用户的模糊需求转化为规划器可直接使用的规范。哪些用于沟通式去幻觉的请求应当向上流转至该角色？
3. 阅读 MetaGPT 第 3 节（“Agents”）。列出 MetaGPT 中 5 个角色各自的输入/输出模式。
4. 阅读 ChatDev 的对话链图（arXiv:2307.07924 Figure 3）。找出沟通式去幻觉机制在何处打破了原本可能陷入无限循环的流程。
5. 普华永道（PwC）实现的 7 倍准确率提升得益于验证循环。请设想三个添加验证器也无济于事的任务场景——即无法进行确定性正确性检查，或检查成本高得令人望而却步的场景。

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------------|------------------------|
| 角色专业化 (Role Specialization) | “不同智能体，不同分工” | 针对规划器 (Planner)、执行器 (Executor)、评审员 (Critic) 和验证器 (Verifier) 等角色专门调优的独立系统提示词 (System Prompts)。 |
| SOP 模式 (SOP Pattern) | “编码化的标准作业程序” | MetaGPT 的架构理念：每个角色遵循严格的输入/输出 (I/O) 模式，将团队协作转化为流水线作业。 |
| 沟通式去幻觉 (Communicative Dehallucination) | “先问再编” | ChatDev 模式：当缺少细节时，执行器会向规划器询问，而不是凭空捏造。 |
| 评审员 (Critic) | “大语言模型审稿人” | 主观且带有明确偏好的评审角色。负责捕捉代码风格与架构偏好问题。但可能被看似合理的文本所欺骗。 |
| 验证器 (Verifier) | “确定性检查” | 基于代码的通过/失败判定。例如测试运行器、类型检查器、模式验证器。无法被欺骗。 |
| 验证缺口 (Verification Gap) | “没人检查过” | 占多智能体系统分类体系 (MAST) 失败案例的 21.3%。答案在未经过本可捕获该缺陷的检查的情况下就被交付。 |
| 修订循环 (Revision Loop) | “评审员打回重做” | 评审员的拒绝会触发执行器根据反馈重新运行。需要设定迭代/计算预算。 |
| 全 LLM 反模式 (All-LLM Anti-pattern) | “我觉得没问题” | 所有角色均由大语言模型 (LLM) 担任，缺乏确定性检查。典型的 MAST 失败模式。 |
