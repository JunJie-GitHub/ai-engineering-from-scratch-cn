给定一个多智能体系统（Multi-Agent System）及其采样执行轨迹（Execution Trace），执行故障模式审计（Failure-Mode Audit）。

产出内容：

1. **样本构建。** 从生产环境中提取至少 200 条轨迹，在任务类型和时间窗口上进行均匀采样。记录采样方法及潜在的偏差风险。
2. **分类阶段。** 针对每条轨迹，标记为 `success | failure`。对于故障，分配一个 MAST 类别（规范 Specification / 协调 Coordination / 验证 Verification），并在适用时附加一个或多个群体思维（Groupthink）家族标签（单一文化 Monoculture / 从众 Conformity / TOM / 混合动机 Mixed-Motive / 级联 Cascade）。
3. **分布表。** 按 MAST 类别和群体思维标签统计数量与百分比。与 Cemri 2025 的参考分布（41.77 / 36.94 / 21.30）进行对比。严重偏离参考分布的系统通常存在特定的薄弱层。
4. **主要故障模式。** 识别出现频率最高的 3 种具体模式（例如，“两个智能体同时执行审查”）。记录复现步骤。
5. **缓解措施排序。** 针对每种主要模式，从标准库中提出缓解方案：显式角色契约、版本化共享状态、独立验证器、熔断器（Circuit Breaker）、检测-诊断-验证（STRATUS）三元组。根据模式出现频率及预期故障减少量进行排序。
6. **静默故障风险。** 有多少故障会产生看似合理但错误的输出，而非引发明显报错？静默故障（Silent Failure）率决定了在验证层（Verification Layer）上的投入力度。
7. **慢速故障代理指标。** 推荐 2-3 个实时指标，以便在故障演变为明显报错前暴露出系统漂移：一致率、重试率、输出长度分布、智能体间编辑距离。

硬性拒绝条件：

- 未采用随机或分层抽样的审计。人工挑选的故障会过度放大极端案例，并遗漏慢速故障漂移。
- 缺乏基线测量的缓解建议。在未知当前故障率的情况下，“添加验证器”毫无意义。
- 忽略 MAST 未知事件。若某条轨迹无法归入现有类别，说明分类体系尚不完整；应提出扩展方案，而非强行归类。
- 在缺乏运营级慢速故障监控的情况下，声称季度审计已足够。季度审计会遗漏两次审计之间的系统漂移。

拒绝规则：

- 若轨迹缺乏按智能体归因的信息（谁生成了什么、谁读取了什么），审计将无法区分协调故障与角色冲突。建议在重新审计前增加结构化的按智能体日志记录。
- 若系统总故障轨迹少于 50 条，则样本量过小，无法生成可靠的分布估计。建议延长观测窗口。
- 若轨迹包含个人身份信息（PII），需在分析前进行脱敏处理。

输出：一份三页的报告。以一句话摘要开头（“41% 规范故障，12% 协调故障，39% 验证缺失，8% 未知；主要模式为双审查者冲突；投资回报率最高的缓解措施为显式角色契约。”），随后展开上述七个部分。最后附上优先级行动清单：列出三项缓解措施，并附带预估实施成本与预期故障率降低幅度。
