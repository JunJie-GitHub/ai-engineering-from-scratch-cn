给定一个由 N 个智能体（Agent）组成的集成系统（Ensemble）回答同一问题，请设计一种共识协议，使其能够抵御三种典型的大语言模型智能体攻击：拜占庭谎言（Byzantine Lie）、阿谀奉承式从众（Sycophantic Conformity）以及相关性错误单一文化（Correlated-Error Monoculture）。

输出内容需包含：

1. **聚类策略（Clustering Strategy）。** 答案如何分组？采用字符串规范化（小写化 + 去除标点）、带阈值的嵌入相似度（Embedding Similarity），还是显式结构规范化（JSON Schema）？说明预期的聚类粒度错误率。
2. **加权策略（Weighting Strategy）。** 采用多数决（Plurality，即计数）、置信度探测加权（CP-WBFT）、质量加信任（WBFT），还是基于分数且具备几何中位数鲁棒性（Geometric-Median Robustness，参考 DecentLLMs）的方案？请根据攻击特征说明选择理由。
3. **阈值（Threshold）。** 达到总权重的多少比例时触发接受？低于阈值时如何处理：重试、升级还是弃权？
4. **多样性要求（Diversity Requirement）。** 集成系统需要多少基础模型、提示词家族或温度设置？单一文化（Monoculture）是多数决无法恢复的攻击类型；多样性是结构化的缓解手段。
5. **独立验证器（Independent Verifier）。** 是否存在只读智能体用于获取真实答案（Ground Truth，若可用）或应用评分标准？验证器的输出流向何处？其结果绝不能重新进入投票池。
6. **轮次限制（Round Bounding）。** 升级前的最大轮数。大多数任务默认为 2-3 轮。轮次过长会加剧阿谀奉承（Sycophancy）现象。
7. **攻击测试表（Attack-Test Table）。** 针对（拜占庭、阿谀奉承、单一文化）每种攻击，展示预期的协议行为与残余风险。若协议存在已知的失效模式，请用一句话说明。

硬性拒绝条件：

- 任何仅对单一基础模型采用多数决的设计。单一文化会导致此类设计静默失效。
- 任何轮次无上限或“持续辩论直至达成一致”的设计。这会奖励从众行为。
- 任何将验证器输出反馈回投票池的设计。这会污染验证器。
- 声称拜占庭容错（BFT）能“解决”分歧的说法。BFT 仅用于对齐输出；正确性是另一个独立的问题。

拒绝规则：

- 若任务无真实答案（如观点、综合、创意类），请明确指出，并建议“共识仅作参考，人类负责决策”。
- 若可用智能体少于 3 个，则不适用共识机制；建议改用“单智能体 + 验证器”方案。
- 若所有智能体共享同一基础模型且用户无法更改，请明确标注单一文化上限。

输出格式：一份单页设计简报。以单句摘要开头（例如：“基于 5 个智能体（3 个基础模型）的置信度加权投票，语义聚类阈值 0.55，独立验证器重新获取来源，最多 2 轮。”），随后依次列出上述七个部分。最后附上攻击测试表。
