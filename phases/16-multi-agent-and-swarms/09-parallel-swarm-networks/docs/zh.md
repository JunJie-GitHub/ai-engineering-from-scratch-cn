# 并行 / 群体（Swarm）/ 网络化架构

> 与监督者（Supervisor）模式对比：无中央决策节点。智能体（Agent）读取共享事件总线（Event Bus），异步领取任务并将结果写回。LangGraph 明确支持用于去中心化、动态环境的“群体架构（Swarm Architecture）”。Matrix（arXiv:2511.21686）将控制流与数据流均表示为通过分布式队列传递的序列化消息，以消除编排器（Orchestrator）瓶颈。其权衡十分明确：以牺牲确定性（Determinism）与可追溯性（Traceability）来换取可扩展性（Scalability）。群体架构适用于包含大量独立子问题的任务；不适用于需要单一连贯规划的任务。

**类型：** 学习 + 构建
**语言：** Python（标准库，`threading`，`queue`）
**前置条件：** 第 16 阶段 · 05（监督者模式），第 16 阶段 · 04（基础模型）
**耗时：** 约 75 分钟

## 问题

监督者模式仅能扩展至少量工作节点。若需扩展至数百个呢？监督者本身就会成为瓶颈：关于任务分配的所有决策都必须汇聚到单一智能体。一个缓慢的规划步骤就会导致整个系统停滞。

群体架构则彻底颠覆了这一设计。工作节点不再依赖中央规划器分发任务，而是从共享队列中主动领取任务。“协调”机制被内置于事件总线的语义中。无需编排器；系统的扩展能力仅受限于队列本身的性能。

## 概念

### 架构形态

                ┌──── shared queue ────┐
                │                      │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     Worker  Worker  Worker   Worker
      A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
            results pool

无编排器（Orchestrator）。每个工作节点（Worker）循环执行以下操作：拉取任务、处理任务、写入结果（并可选择将后续任务加入队列）。

### 适用场景

- **大量独立任务。** 例如数据抓取（Scraping）、数据转换（Transforming）、数据分类（Classifying）。任务之间互不依赖。
- **耗时差异较大的工作。** 如果某些任务耗时 100 毫秒，而其他任务耗时 10 秒，集群（Swarm）会自动实现负载均衡——处理速度快的节点会自动拉取下一个任务。而监督模式（Supervisor）则需要提前预估任务耗时。
- **吞吐量优先于确定性。** 你更关注整体完成时间，而非严格的执行顺序。

### 失效场景

- **有序工作流。** 如果步骤 3 依赖步骤 2 的输出，集群模式可能导致步骤 3 在步骤 2 完成前就被触发。
- **需要全局规划的任务。** 复杂的研究问题更适合由规划器（Planner）统筹。一群独立工作的“研究员”只会产出零散的事实，无法生成连贯的报告。
- **调试困难。** 由于缺乏集中式日志且工作异步进行，复现 Bug 的成本极高。

### Matrix 架构 (arXiv:2511.21686)

Matrix 是 2025 年的一篇论文，它将集群模式推向了极致：控制流（Control Flow）和数据流（Data Flow）均作为序列化消息在分布式队列中传递。系统没有中央协调器。容错能力依赖于消息的持久化（Message Durability）。而可扩展性（Scalability）则交由消息代理（Message Broker）解决，而非系统本身。

核心贡献：提出了一种编程模型，将多智能体（Multi-Agent）协作转化为“该智能体订阅了哪个消息主题（Message Topic）？”，而非“监督器下一步该选择哪个智能体？”。这使得整个系统呈现出发布/订阅（Pub/Sub）事件网格（Event Mesh）的形态。

### LangGraph 的集群架构

LangGraph 2025 版文档明确将“集群架构（Swarm Architecture）”列为多智能体模式之一：智能体作为节点，但边构成带有环路的有向图（Directed Graph），且池中的任何节点均可被激活。工作节点根据条件从可用任务中自行拉取，而非由监督器分配。

### 失效模式：饥饿与热点倾斜

如果所有工作节点都优先拉取当前最快的可用任务，长耗时任务将永远得不到处理，直到队列中只剩下它们。这是典型的队列饥饿（Queue Starvation）问题。

缓解策略：
- 采用显式老化机制的优先级队列（Priority Queue）（随等待时间增加优先级）。
- 工作节点专业化：部分节点仅处理“长耗时”任务。
- 背压机制（Back-pressure）：限制快速任务进入队列的数量。

### 与基于内容路由的关联

集群模式与基于内容的路由（Content-Based Routing，第 22 课）天然契合。与其使用通用队列，不如为每种消息类型设置独立队列。专业型工作节点仅订阅其对应的消息类型。这正是消息总线（Message Bus）架构能够扩展至数千个智能体的基础。

## 动手构建

`code/main.py` 实现了一个由 4 个工作线程 (worker threads) 组成的蜂群模式 (swarm)，它们从共享的 `queue.Queue` 中拉取任务。任务的执行时长各不相同（有的快，有的慢）。该演示对比了以下三种模式：

- **顺序基线 (Sequential baseline)：** 单个工作线程串行处理所有任务。
- **固定分配 (Fixed assignment)：** 每个任务预先分配给特定的工作线程（主管模式 (supervisor-style)）。
- **蜂群 (Swarm)：** 工作线程从共享队列中自主拉取任务。

蜂群模式能自动实现负载均衡；而固定分配模式下，当分配给某个工作线程的任务较慢时，其他处理速度快的工作线程就会处于空闲状态。

运行方式：

python3 code/main.py

输出结果将显示每个工作线程处理的任务数量（蜂群模式的分配虽不均衡，但达到了最优）以及挂钟时间 (wall-clock time)。

## 使用指南

`outputs/skill-swarm-fit.md` 用于评估某项任务应采用蜂群模式还是主管模式。评估输入包括：任务独立性、时长方差、顺序要求以及可调试性需求。

## 部署上线

检查清单：

- **带老化机制的优先级队列 (Priority queue with aging)。** 防止长任务饥饿 (starvation)。
- **工作线程幂等性 (Worker idempotency)。** 如果工作线程在运行中途崩溃，任务可能会被重复拉取。因此，工作线程必须具备幂等性。
- **持久化队列 (Durable queue)。** 生产环境请使用 Kafka、Redis Streams 或基于数据库的队列。`queue.Queue` 仅为内存队列。
- **任务级可观测性 (Observability per task)。** 每个任务都分配有追踪 ID (trace ID)；每个工作线程在任务开始和结束时都会记录该 ID。
- **背压机制 (Back-pressure)。** 如果队列的增长速度超过工作线程的处理速度，需降低生产者的生成速率。

## 练习

1. 运行 `code/main.py`。在变长任务负载下，蜂群模式比顺序模式快多少？比固定分配模式快多少？
2. 添加优先级队列变体（使用 `queue.PriorityQueue`）。根据任务的“重要性 (importance)”字段分配优先级。观察在持续负载下，低优先级任务是否会出现饥饿现象。
3. 实现热点检测器 (hot-spot detector)：当任意工作线程处理的任务数量达到最慢工作线程的 3 倍时，记录日志。这反映了任务时长分布的什么特征？
4. 阅读 Matrix 论文 (arXiv:2511.21686) 的摘要和第 3 节。指出 Matrix 接受的一项具体权衡（可扩展性提升）以及放弃的一项特性（可追溯性 (traceability)、确定性 (determinism)）。
5. 将蜂群演示改造为使用包含 `(task_type, payload)` 元组的 `queue.Queue`，并让工作线程仅订阅特定类型的任务。当任务具有异构性 (heterogeneous) 时，采用何种路由规则更为合理？

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------------|------------------------|
| 蜂群架构 (Swarm Architecture) | “去中心化智能体” | 工作节点 (Worker) 从共享队列中拉取任务；无中央编排器 (Orchestrator)。 |
| 事件总线 (Event Bus) | “智能体订阅主题” | 根据任务类型或内容将任务路由至工作节点的消息代理 (Message Broker)。 |
| 饥饿 (Starvation) | “任务永远无法执行” | 由于高优先级任务持续到达，低优先级任务始终无法被调度执行。 |
| 热点倾斜 (Hot-spotting) | “单个工作节点不堪重负” | 负载不均衡，导致大部分任务集中分配给单个工作节点。 |
| 背压 (Back-pressure) | “让生产者减速” | 当队列满载时，向上游发送信号以停止生产任务的机制。 |
| 幂等工作节点 (Idempotent Worker) | “可安全重试” | 任务被重复处理两次仍产生相同结果。由于工作节点可能在运行中途崩溃，此特性必不可少。 |
| 持久化队列 (Durable Queue) | “崩溃后数据不丢失” | 基于磁盘或副本存储的队列；工作节点崩溃时任务不会丢失。 |
| Matrix 框架 (Matrix Framework) | “全消息传递蜂群” | 数据流与控制流均作为序列化消息在分布式队列中传递。 |
