给定任务及其吞吐量 (Throughput) / 确定性 (Determinism) 要求，推荐采用群体 (Swarm) 架构或主管 (Supervisor) 架构，并列出具体的队列 (Queue) 与护栏 (Guardrail) 选择。

输出内容：

1. **任务独立性检查。** 子任务是相互独立还是彼此依赖？仅在独立性较高时适用群体架构。
2. **耗时分布。** 均匀分布与可变分布。群体架构在处理耗时波动较大的工作负载时更具优势。
3. **顺序要求。** 严格、宽松或无要求。群体架构不保证执行顺序；主管架构则能保持顺序。
4. **可调试性需求。** 高（金融、医疗领域）→ 主管架构。中 → 群体架构配合单任务追踪 ID (Trace IDs)。
5. **队列选择。** 演示环境使用内存队列（`queue.Queue`）；生产环境使用 Kafka / Redis Streams / NATS / 基于持久化数据库的队列。
6. **工作节点设计要求。** 必须具备幂等性 (Idempotency)；必须输出单任务追踪日志；必须能够处理背压 (Back-pressure)。
7. **防饥饿策略 (Anti-starvation)。** 优先级老化 (Priority Aging)、工作节点专业化、有界队列 (Bounded Queue)。
8. **可观测性方案 (Observability)。** 单任务 ID、开始/结束事件、结果池模式。

硬性否决条件：

- 对具有严格顺序要求的任务推荐群体架构。
- 工作节点不具备幂等性的群体架构。
- 生产环境中未使用持久化队列的群体架构。

拒绝规则：

- 若任务每秒独立单元数少于 10 个，则拒绝群体架构并推荐主管架构。在低吞吐量场景下，群体架构的额外开销得不偿失。
- 若可观测性要求需要单一连贯的追踪链路（如审计、合规），则拒绝群体架构，并推荐改用 LangGraph 确定性图 (Deterministic Graph)。

输出格式：一页纸架构简报。开篇给出适配结论，结尾针对目标吞吐量给出具体的消息代理 (Message Broker) 推荐。
