给定多智能体代码库或架构文档，审计其共享内存设计，并标记内存投毒的暴露风险。

输出内容：

1. **拓扑结构（Topology）。** 采用完整消息池、按主题分区的黑板（blackboard）、面向各智能体的投影视图，还是混合架构？指明所使用的数据结构（如列表、字典、pandas DataFrame、向量数据库、SQL 表）。估算稳态下写入者与读取者数量的粗略上限。
2. **溯源字段（Provenance fields）。** 每次写入时，条目是否记录：写入者 ID、时间戳、提示词哈希值或提示词原文、工具调用轨迹、来源 URI 或工具名称？列出已包含的字段与缺失的字段。
3. **更新模型（Update model）。** 日志是仅追加（append-only）模式，还是允许写入者就地修改（mutate in place）？若允许修改，采用何种并发控制机制（锁、乐观版本控制、无）？修正操作应以替代条目（supersession entries）的形式记录，而非就地编辑——标记任何未遵循此原则的设计。
4. **验证器隔离（Verifier separation）。** 是否存在具备独立数据源访问权限的只读智能体？它能否向主池写入数据（不应允许）？其输出结果流向何处？
5. **投影模式（Projection schema）。** 若设计采用投影机制（如 LangGraph 的 reducer、黑板主题、角色作用域视图），该模式是否有文档说明？新智能体如何声明其消费的投影？
6. **投毒风险评分（Poisoning risk score）。** 在以下各维度进行 1-5 分评分：[溯源完整性]、[替代优于就地修改]、[验证器独立性]、[投影模式清晰度]。若系统在任一维度得分低于 3 分，则予以标记。

硬性拒绝标准（Hard rejects）：

- 任何未标记“缺失验证器”的审计报告。具备独立数据源访问权限且不可写入的验证器是核心缓解机制（load-bearing mitigation）；若无此机制，其他所有缓解手段皆为装饰。
- 建议“增加更多测试”的审计。测试无法捕获内存投毒，因为投毒生成的输出通常看似合理，能够通过测试。
- 建议仅将内容哈希作为唯一溯源依据的审计。哈希值只能告诉你*写了什么*，无法说明*是谁写的*或*来自何处*。

拒绝执行规则（Refusal rules）：

- 若代码库将共享状态隐藏于外部服务（如 Redis、Postgres、向量数据库）中且缺乏检查工具，需声明：若无生产环境读取权限，审计无法完成。
- 若系统包含的智能体少于三个，需注明：内存投毒风险较低，但建立溯源机制仍是低成本的有效保障。
- 若系统使用内置状态管理框架（如 LangGraph checkpointer、AutoGen pool），应直接审计该框架提供的保障机制，而非重新推导。

输出要求：一份两页的报告。以一句话摘要开头（例如：“共享状态采用完整消息池，缺乏溯源机制与验证器——投毒风险极高。”），随后依次展开上述六个部分。最后附上按优先级排序的行动清单：列出三项改进措施，每项标注 [critical]（关键）、[should]（建议）或 [nice-to-have]（可选），并附带预估实施时间。
