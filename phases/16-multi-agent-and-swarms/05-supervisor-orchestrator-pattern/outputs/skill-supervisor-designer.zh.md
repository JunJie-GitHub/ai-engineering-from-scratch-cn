针对适合并行子智能体（subagent）研究的用户查询，生成一份监督者模式（supervisor-pattern）设计方案，该方案可直接接入任意框架（如 LangGraph、OpenAI Agents SDK、CrewAI Hierarchical）。

输出内容：

1. **复杂度评估。** 该查询属于简单（1个智能体，3-10次工具调用）、中等（2-4个工作者）还是复杂（5个以上工作者）？请结合 Anthropic 的规模-工作量启发式规则（scale-effort heuristic），用一句话说明理由。
2. **主导系统提示词（lead system prompt）。** 必须包含：(a) 任务分解指令，(b) 结果综合指令，(c) 明确规则：主导智能体绝不直接阅读原始源内容，仅处理工作者提交的摘要。
3. **工作者系统提示词。** 每个角色对应一份，需明确界定其狭窄的职责范围，以及主导智能体所期望的输出格式。
4. **子问题分解规则。** 主导智能体如何拆分查询？采用“先宽后窄”策略还是直接分解？哪些情况会导致子问题不合格（如与其他子问题重叠、范围过宽）？
5. **综合模板。** 明确的冲突处理规则：若两名工作者返回相互矛盾的事实，综合阶段必须明确指出分歧，而非静默选择其一。
6. **模型搭配。** 主导智能体使用何种模型（推理层级），工作者使用何种模型（更快/更便宜层级）。请说明其中的权衡取舍。
7. **可观测性要求。** 最低追踪节点：计划阶段、每个工作者的开始/结束、综合输入、综合输出。

硬性拒绝条件：

- 任何主导智能体自行使用工具的设计。主导智能体仅负责规划与综合。
- 允许范围漂移的工作者提示词（例如“研究与X相关的任何内容”且未设定边界）。
- 掩盖冲突的综合模板。

拒绝规则：

- 若查询较为简单（预估总工具调用次数低于10次），则拒绝该设计方案，并推荐改用单智能体（single-agent）模式。请引用 Anthropic 关于 15 倍 Token 成本的研究结论。
- 若查询具有顺序依赖性（步骤2需要步骤1的输出），则拒绝该方案，并推荐改用流水线/链式（pipeline/chain）模式。
- 若用户优先追求确定性与可审计性，则拒绝监督者模式，并推荐改用 LangGraph 静态图。

输出格式：一页纸的设计简报。开篇需包含复杂度评估与模式适配结论（“监督者模式适用”）。若系统将持续运行，结尾需附上彩虹部署（rainbow-deployment）提醒。
