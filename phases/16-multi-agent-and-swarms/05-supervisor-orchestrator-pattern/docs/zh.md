# 监督者/编排者-工作者模式 (Supervisor / Orchestrator-Worker Pattern)

> 一个主导智能体（lead agent）负责规划与任务分发；专业的工作者智能体（worker）在并行上下文中执行任务并汇报结果。这正是 Anthropic 研究系统背后的架构模式（由 Claude Opus 4 担任主导，Sonnet 4 作为子智能体），在内部研究评估中，其表现较单智能体（single-agent）Opus 4 提升了 90.2%。Anthropic 的工程博文指出，BrowseComp 基准测试中 80% 的性能差异仅由 Token 使用量（token usage）解释——多智能体架构胜出的核心原因在于每个子智能体都能获得全新的上下文窗口（context window）。本课程将从底层原语（primitives）出发构建监督者模式（supervisor pattern），并涵盖 2026 年生产环境部署中的工程实践经验。

**类型：** 学习 + 实践
**编程语言：** Python（标准库，`threading`）
**前置要求：** 第 16 阶段 · 04（原语模型）
**预计耗时：** 约 75 分钟

## 问题

研究任务是单智能体系统极易失败的经典场景。例如，当你提问“2023 年至 2026 年间多智能体系统发生了哪些变化？”时，单智能体会按顺序阅读五篇论文，其上下文窗口会被这些文本占据大半，随后它必须综合所有内容进行推理。等到处理到第五篇时，它早已遗忘了第一篇的内容。更重要的是，它无法实现并行处理。

监督者模式正是为了解决这一问题：由一个主导智能体规划搜索路径，将每个子问题分发给工作者智能体，最后进行综合汇总。每个工作者智能体都能为特定的细分问题分配独立的 20 万 Token 上下文窗口。主导智能体永远不会看到原始论文——它只接收工作者智能体提交的摘要。

Anthropic 生产环境中的研究系统在内部评估中，表现较单智能体 Opus 4 提升了 90.2%。同一篇博文还指出，BrowseComp 基准测试中 80% 的性能差异仅由 *Token 使用量* 决定。为每个子智能体提供全新的上下文窗口，正是该架构取得优势的核心机制。

## 核心概念

### 模式架构

主管（Lead）从不直接处理原始输入。在主管进行综合汇总前，各工作节点（Worker）也无法看到彼此的工作成果。图中的每个箭头代表一次任务交接，且每次传递的中间产物（artifact）都经过严格限定。

### 核心优势

三大机制：

1. **每个子代理（subagent）拥有独立的上下文。** 当工作节点探索“FIPA-ACL 历史沿革”时，它不会携带主管在规划阶段消耗的 4 万 token。它拥有 20 万 token 的上下文窗口，专门用于解决单一问题。
2. **通过提示词（prompt）实现专业化分工。** 主管的提示词侧重于“拆解与综合”，而非“研究”。每个工作节点的提示词则非常聚焦，例如“查找 X 中的变更”。聚焦的提示词能产出聚焦的结果。
3. **并行处理（Parallelism）。** 各工作节点并发运行。实际耗时（wall-clock time）大致为 `max(worker_times) + plan + synthesis`，而非 `sum(worker_times)`。

### 工程实践经验（Anthropic 2025）

Anthropic 的文章列举了多项生产环境经验，这些经验在 2026 年依然适用：

- **根据查询复杂度动态分配资源。** 简单查询：单智能体（agent），3-10 次工具调用。复杂查询：10 个以上智能体。此项评估必须由主管完成，而非调用方。
- **先广后深。** 首先将问题拆解为宽泛的子问题，如果答案需要进一步深入，再针对每个子问题派生更多工作节点。
- **彩虹部署（Rainbow deployments）。** 智能体通常是长运行且有状态的（stateful），传统的蓝绿部署（blue-green deployment）并不适用。Anthropic 采用彩虹部署：在逐步下线旧版本的同时，渐进式发布新版本。
- **Token 消耗占主导。** 多智能体（multi-agent）架构的 Token 消耗量约为单智能体的 15 倍。仅在任务价值足以覆盖成本时才应启用该架构。

### LangGraph 的架构演进

LangGraph 最初发布了一个 `langgraph-supervisor` 库，其中包含高级辅助函数 `create_supervisor`。到了 2025 年，LangChain 官方建议改为直接通过工具调用（tool-calling）来实现主管模式，因为工具调用能让开发者更精确地控制*主管所能看到的上下文*（上下文工程/context engineering）。该库目前仍可正常使用，但官方文档现已推荐使用工具调用的实现方式。

### 典型失效模式

- **主管规划产生幻觉（hallucination）。** 如果主管生成的子问题未能正确拆解核心问题，工作节点就会在错误的目标上进行精确研究。
- **工作节点过度探索。** 若缺乏明确的范围边界，工作节点可能会偏离其负责的子问题，进而污染后续的综合汇总步骤。
- **综合汇总冲突。** 两个工作节点返回相互矛盾的事实。主管必须选择重新提问（增加一轮交互）或明确标注分歧。静默选择其中一方是最严重的失效：用户将完全不知晓分歧的存在。

### 不适用该模式的场景

- **串行任务。** 如果第二步严格依赖第一步的输出，并行化毫无意义。应使用流水线架构（如 CrewAI Sequential 或 LangGraph 线性图）。
- **简单查询。** 单智能体处理速度更快且成本更低。在派生工作节点前，应先通过主管的“资源分配评估”进行判断。
- **强确定性要求。** 主管模式依赖大语言模型（LLM）动态选择委派路径。当审计与回放（audit/replay）的重要性高于系统适应性时，静态图（static graphs）是更优选择。

## 构建它

`code/main.py` 使用 `threading` 实现了一个包含三个并行工作节点（worker）的监督器（supervisor）。主导节点（lead）将查询分解为子问题，工作节点并发处理每个子问题，最后由主导节点进行综合（synthesize）。此处未使用真实的大语言模型（LLM）——工作节点通过脚本模拟了“获取并总结”的流程。

核心结构：

- `Lead.plan(query)` 将查询拆分为 3 个子问题。
- `Worker.run(sub_q)` 返回模拟的总结结果（在生产环境中可替换为任意使用工具的智能体（agent））。
- `Lead.run(query)` 在线程中启动工作节点，等待其完成（join），并进行综合。

运行方式：

python3 code/main.py

输出结果将展示执行计划、带有起止时间戳的并行工作节点追踪记录（trace），以及最终的综合结果。你可以直观地看到实际耗时（wall-clock time）的优势：三个耗时 0.3 秒的工作节点并行运行仅需约 0.35 秒，而非 0.9 秒。

## 使用它

`outputs/skill-supervisor-designer.md` 接收用户查询并生成监督器模式（supervisor pattern）的设计方案：包括主导节点的系统提示词（system prompt）、工作节点角色、子问题分解规则以及综合模板。在构建新型研究型智能体系统之前，建议先使用此文件进行设计。

## 部署它

部署监督器模式前的检查清单：

- **模型搭配（model pairing）。** 主导节点使用推理级模型（如 Opus 系列或 `o3` 系列）。工作节点使用更快、更经济的模型（如 Sonnet 或 `o4-mini`）。
- **工作节点超时（worker timeout）。** 任何运行时间超过中位数 2 倍的工作节点将被终止；主导节点可选择以更窄的范围重新生成该节点，或在缺少该节点的情况下继续执行。
- **单节点 Token 上限（token cap per worker）。** 设置硬性限制（例如预期综合输入量的 10 倍），可防止失控的工作节点耗尽预算。
- **可观测性（observability）。** 记录主导节点的计划、每个工作节点的工具调用（tool calls）以及综合过程。这是进行事后调试（post-hoc debugging）的基础。
- **渐进式发布（rainbow rollout）。** 有状态的长周期智能体需要逐步进行版本过渡，而非热替换（hot swap）。

## 练习

1. 运行 `code/main.py`，然后修改主导节点逻辑，使其生成 5 个工作节点而非 3 个。观察实际耗时的变化。在此演示中，工作节点数量达到多少时，生成开销会超过并行带来的收益？
2. 实现工作节点超时机制：终止任何运行超过 0.5 秒的工作节点，并让主导节点综合剩余结果。你需要哪些可观测性数据来确认某个工作节点已被终止？
3. 在主导节点的综合步骤中添加冲突检测：如果两个工作节点返回相互矛盾的答案，主导节点应记录分歧而非直接选择其一。如何在不调用大语言模型的情况下检测矛盾？
4. 阅读 Anthropic 关于研究系统工程的博文。列出本示例演示若要投入生产环境所需采纳的三项实践。
5. 对比 LangGraph 的 `create_supervisor`（旧版）与新的工具调用（tool-calling）推荐方案。哪种方案能让你更好地控制监督器所看到的内容？为何 Anthropic 明确建议仅将子答案传入综合步骤，而非原始的工作节点上下文？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------------|------------------------|
| 主管智能体 (Supervisor) | “主导智能体” | 负责规划、委派与综合的编排智能体 (Orchestrator Agent)。其本身不执行具体任务。 |
| 工作智能体 (Worker) | “子智能体” | 由主管智能体调用的专注型智能体，具有明确的作用域和独立的上下文窗口 (Context Window)。 |
| 编排-工作模式 (Orchestrator-worker) | “主管模式” | 同一概念的不同称呼。2026 年的文献中两者均有使用。 |
| 全新上下文 (Fresh Context) | “干净窗口” | 工作智能体的上下文从其系统提示词 (System Prompt) 和分配的问题开始，而非主导智能体的历史记录。 |
| 彩虹部署 (Rainbow Deployment) | “渐进式发布” | 长期运行的有状态智能体 (Stateful Agent) 需要基于版本的排空与替换 (Drain-and-Replace) 策略，而非蓝绿部署 (Blue-Green Deployment)。 |
| Token 主导效应 (Token Dominance) | “上下文即变量” | 据 Anthropic 研究，研究评估中 80% 的方差来源于使用的 Token 总量，而非模型选择。 |
| 规模动态调整 (Scale Effort) | “智能体数量匹配复杂度” | 主导智能体评估查询难度，据此动态生成 1 个或 10 个以上的工作智能体。 |
| 综合冲突 (Synthesis Conflict) | “工作智能体意见分歧” | 两个工作智能体返回相互矛盾的事实；主导智能体必须明确暴露分歧，而非静默选择其一。 |
