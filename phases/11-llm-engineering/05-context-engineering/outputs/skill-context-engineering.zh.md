# 上下文工程（Context Engineering）

在构建大语言模型（Large Language Model, LLM）应用时，可应用此框架来设计上下文组装流水线。

## 核心原则

1. **上下文资源稀缺。** 128K 的上下文窗口（Context Window）听起来很大，但很快就会填满。必须为每个组件明确分配预算。
2. **注意力分布不均。** 模型对开头和结尾的关注度更高。应将关键信息放置在这些位置。中间区域是“注意力盲区”。
3. **动态优于静态。** 不同的查询需要不同的上下文。应按查询动态组装，而非在启动时一次性组装。
4. **少即是多。** 精心筛选的 10K 上下文优于直接堆砌的 100K 上下文。信噪比比信息总量更重要。
5. **量化一切。** 无法衡量就无法优化。在每次请求中统计每个组件的词元（Token）消耗。

## 上下文预算指南

| 组件 | 典型范围 | 优先级 | 压缩策略 |
|-----------|-------------|----------|---------------------|
| 系统提示词（System Prompt） | 200-1,000 词元 | 固定，高 | 精简编写，去除冗余 |
| 工具定义（Tool Definitions） | 500-3,000 词元 | 动态，中 | 根据查询意图裁剪 |
| 检索上下文（Retrieved Context） | 1,000-5,000 词元 | 动态，高 | 重排序 + 阈值过滤 + 去重 |
| 对话历史（Conversation History） | 500-5,000 词元 | 动态，中 | 总结早期对话轮次 |
| 少样本示例（Few-shot Examples） | 500-2,000 词元 | 动态，高 | 按任务相似度筛选 |
| 用户查询（User Query） | 50-500 词元 | 固定，最高 | 不适用 |
| 生成预留空间（Generation Reserve） | 2,000-8,000 词元 | 固定 | 根据预期输出长度调整 |

## 何时使用各类记忆机制

**短期记忆（对话历史）：** 针对当前会话。通过摘要进行管理。压缩超过 5-10 轮交互的旧对话。保留最近 3-4 轮对话的原始内容。

**长期记忆（事实数据库）：** 跨会话持续存在的偏好与项目事实。在会话开始时检索。示例：“用户偏好 Python”、“项目使用 PostgreSQL”、“团队遵循主干开发模式”。可存储在 `CLAUDE.md`、数据库或结构化记忆系统中。

**情景记忆（过往交互）：** 与当前任务相关的具体历史对话。以嵌入向量（Embeddings）形式存储，通过相似度检索。例如“上周我们调试过类似的认证问题”即属于情景记忆。

## 工具选择策略

不要在每次请求中包含所有工具。这会浪费词元并干扰模型判断。

1. 分类查询意图（代码、邮件、日历、研究、数据）
2. 将意图映射到对应的工具类别
3. 仅包含匹配的工具
4. 若意图模糊，则包含排名前 2 类别的工具
5. 始终包含一个“通用”工具（如网络搜索）作为后备方案

预期节省：在意图明确的查询中，可节省 60-80% 的工具定义词元。

## 检索最佳实践

- **检索后重排序。** 向量相似度仅为粗略过滤。使用重排序器（Reranker，如交叉编码器或基于 LLM 的模型）可显著提升精确率。
- **设置相关性阈值。** 不要包含余弦相似度低于 0.3 的文本块（Chunks）。它们只会增加噪声。
- **去重处理。** 若两个文本块有 80% 以上的内容重合，仅保留得分较高的一个。
- **应用“中间迷失”（Lost-in-the-Middle）排序策略。** 将最相关的文本块放置在开头和结尾。
- **限制检索总词元数。** 3-5 个高度相关的文本块优于 15 个质量平平的文本块。

## 历史对话管理

- 保留最近 3-4 轮对话的原始内容（模型需要近期上下文）
- 将较早的对话轮次总结为摘要（例如：“我们讨论了 X，决定了 Y，并在 Z 处受阻”）
- 剔除不增加有效信息的系统生成轮次（如无面向用户内容的工具调用记录）
- 当历史记录超过可用预算的 30% 时触发压缩机制

## 警示信号

- 系统提示词（System Prompt）超过 2,000 个词元（Tokens）：可能包含了本应动态处理的信息
- 每次请求均包含所有工具（Tools）：应实现基于意图的选择（Intent-based Selection）
- 检索（Retrieval）阶段缺乏相关性过滤（Relevance Filtering）：相当于向上下文窗口（Context Window）中注入噪声
- 对话历史（Conversation History）无限增长：未实现摘要（Summarization）机制
- 未预留生成空间（Generation Reserve）：导致模型截断（Truncate）其响应
- 相同信息重复出现在三个位置（系统提示词、检索文档、历史记录）：需进行去重（Deduplicate）处理
- 上下文利用率（Context Utilization）超过 60%：留给模型“思考”的空间过少
