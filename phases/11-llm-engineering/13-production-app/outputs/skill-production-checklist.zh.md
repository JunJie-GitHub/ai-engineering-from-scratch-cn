# 生产环境大语言模型（LLM）检查清单

在将大语言模型（LLM）应用交付上线时，请按顺序逐项核对本清单。每个部分均包含带有具体阈值的通过/失败标准。

## 1. 安全性（发布阻断项）

在部署任何版本前，此处每一项都必须通过。

| 检查项 | 通过标准 | 验证方法 |
|-------|--------------|---------------|
| API 密钥置于环境变量中 | 代码库中零硬编码密钥 | `grep -r "sk-" --include="*.py"` 无返回结果 |
| 输入防护机制已启用 | 提示词注入（Prompt Injection）模式被拦截 | 发送 "Ignore all previous instructions" -- 返回拦截响应 |
| 个人身份信息（PII）脱敏 | 捕获社保号、信用卡、邮箱等模式 | 发送 "My SSN is 123-45-6789" -- 在调用 LLM 前完成 PII 脱敏 |
| 输出过滤 | 拦截危险内容 | 模型无法返回 `DROP TABLE`、`rm -rf`、`exec()` 等模式 |
| 速率限制（Rate Limiting） | 强制执行单用户请求上限 | 同一用户 10 秒内发送 100 次请求 -- 最后 50+ 次被拒绝 |
| 所有端点均需认证 | 禁止未认证的 LLM 访问 | `curl /v1/chat` 不带 token 返回 401 |
| 限制跨域资源共享（CORS） | 仅允许生产环境域名 | `Origin: evil.com` 请求被拒绝 |
| 最大输入 Token 数 | 拒绝超限请求 | 发送 50K token 输入 -- 返回 413 或触发截断 |

## 2. 可靠性（首周生存保障）

这些措施可避免你遭遇首次值班事故（On-call Incident）。

| 检查项 | 通过标准 | 验证方法 |
|-------|--------------|---------------|
| 带退避的重试机制 | 遇到 5xx 错误重试 3 次，采用指数退避延迟 | 在请求中途终止 LLM 模拟服务 -- 日志中可见重试记录 |
| 备用模型链 | 链路中包含 2 个及以上模型 | 主模型不可用 -- 仍能从备用模型获取响应 |
| 请求超时设置 | 所有外部调用最长 30 秒 | 使用慢速 LLM 模拟服务（60 秒） -- 请求在 30 秒时超时 |
| 优雅降级（Graceful Degradation） | 缓存或检索增强生成（RAG）故障不会导致服务崩溃 | 停止缓存服务 -- 请求仍能成功（速度变慢、成本增加） |
| 健康检查端点 | 返回依赖项状态 | `GET /health` 返回 `{"status": "healthy", "cache": ..., "llm": ...}` |
| 流式输出正常 | 首字延迟（Time-to-First-Token）低于 500 毫秒 | 测量首字延迟，持续稳定 < 500 毫秒 |
| 错误信息安全 | 内部错误绝不泄露给用户 | 强制触发 500 错误 -- 用户仅看到通用错误提示，而非堆栈跟踪（Stack Trace） |

## 3. 成本控制（首月经济性）

这些措施可防止收到 5 万美元的意外账单。

| 检查项 | 通过标准 | 验证方法 |
|-------|--------------|---------------|
| 追踪单次请求成本 | 每次请求均记录 Token 数量与美元成本 | 请求日志包含 `input_tokens`、`output_tokens`、`cost_usd` 字段 |
| 语义缓存（Semantic Cache）已启用 | 重复模式命中率 > 20% | 1000 次测试请求后，缓存统计显示命中率 |
| 配置缓存生存时间（TTL） | 条目会过期（默认：1 小时） | 插入条目 -- 超过 TTL 后不再返回 |
| 按用户追踪成本 | 按 `user_id` 聚合成本 | 仪表盘/API 显示成本排名前 10 的用户 |
| 成本告警 | 达到每日预算 80% 时触发告警 | 设置每日 10 美元预算，发送价值 8.50 美元的请求 -- 触发告警 |
| 基于成本的路由 | 低复杂度查询使用更便宜的模型 | 简单问题路由至 `gpt-4o-mini`，复杂问题路由至 `gpt-4o` |
| 设置最大输出 Token 数 | 按模板限制响应长度 | 模板设置 `max_output_tokens=512` -- 响应绝不超出该限制 |

**成本估算公式：**
Monthly LLM cost = DAU x queries_per_user x 30 x (1 - cache_hit_rate) x (avg_input_tokens x input_price + avg_output_tokens x output_price) / 1,000,000

**按规模划分的基准阈值：**

| 日活跃用户数（DAU） | 目标单次请求成本 | 月度预算 |
|-----|-------------------|----------------|
| 1K | < $0.005 | < $750 |
| 10K | < $0.003 | < $4,500 |
| 100K | < $0.001 | < $15,000 |

## 4. 可观测性（生产环境调试）

无法观测，便无从修复。

| 检查项 | 通过标准 | 验证方法 |
|-------|--------------|---------------|
| 结构化 JSON 日志记录 (Structured JSON Logging) | 每个请求生成一行 JSON 日志 | 日志包含：request_id, user_id, model, tokens, latency_ms, cost |
| 请求追踪 (Request Tracing) | 包含各组件耗时的端到端追踪链路 | 单个请求显示：guardrail (5ms) + cache (2ms) + llm (3200ms) + eval (1ms) |
| 延迟监控 (Latency Tracking) | 测量 P50、P95、P99 延迟指标 | 1000 次请求后：P50 < 2s，P99 < 10s |
| 错误率监控 (Error Rate Monitoring) | 错误被计数并分类 | 仪表盘显示：0.5% API 错误，0.1% 护栏拦截，0.01% 超时 |
| 缓存指标 (Cache Metrics) | 可见命中率、未命中率及缓存条目数 | `GET /v1/cache/stats` 返回当前数值 |
| A/B 测试指标 (A/B Test Metrics) | 记录每个实验变体的质量指标 | 每个请求记录 prompt_template + 版本以供对比 |
| 评估日志记录 (Eval Logging) | 每个请求记录质量信号 | 存储响应长度、延迟、模型、模板版本用于离线分析 |

## 5. 提示词管理 (Prompt Management)

提示词即代码。请像对待代码一样对待它们。

| 检查项 | 通过标准 | 验证方法 |
|-------|--------------|---------------|
| 模板版本控制 (Versioned Templates) | 每个模板均包含名称 + 版本字符串 | 模板变更会创建新版本，旧版本保留 |
| A/B 测试支持 (A/B Testing Support) | 通过确定性用户哈希进行流量分割 | 同一用户在实验期间始终看到相同的变体 |
| 回滚能力 (Rollback Capability) | 可在 1 分钟内回退至上一版本 | 修改实验配置后，流量立即切换 |
| 模板校验 (Template Validation) | 渲染前对变量进行校验 | 模板中缺失变量会抛出明确错误，而非 `KeyError` |
| 系统提示词分离 (System Prompt Separation) | 系统消息与用户消息分属不同字段 | 系统提示词不会拼接至用户消息中 |

## 6. 扩展就绪性 (Scaling Readiness)

上线初期无需具备。业务规模增长 10 倍时必不可少。

| 检查项 | 通过标准 | 验证方法 |
|-------|--------------|---------------|
| 异步 LLM 调用 (Async LLM Calls) | API 调用不阻塞线程 | 50 个并发请求下，服务器 CPU 占用率 < 30% |
| 连接池 (Connection Pooling) | 复用 HTTP 连接 | 网络抓包显示与大语言模型提供商保持持久连接 |
| 水平扩展 (Horizontal Scaling) | 无状态服务器设计 | 负载均衡器后部署 2 个实例，所有请求均成功 |
| 队列支持 (Queue Support) | 非实时任务进入消息队列 | 摘要请求返回 job_id，结果可通过轮询获取 |
| 负载测试 (Load Testing) | 100 并发用户，错误率 < 5% | 在目标并发量下，`wrk` 或 `locust` 测试通过 |

## 新项目实施顺序

1. **第 1 天：** API 服务器 + 提示词模板 + 带重试机制的单一 LLM 调用
2. **第 2 天：** 输入护栏 (Input Guardrails) + 输出护栏 + 错误处理
3. **第 3 天：** 语义缓存 (Semantic Cache) + 单次请求成本追踪
4. **第 4 天：** 流式传输 (Streaming, SSE) + 健康检查端点
5. **第 5 天：** 结构化日志 + 请求追踪 + 评估日志
6. **第 2 周：** A/B 测试 + 提示词版本控制 + 回滚机制
7. **第 3 周：** 备用模型链 (Fallback Model Chain) + 优雅降级 (Graceful Degradation)
8. **第 4 周：** 负载测试 + 异步优化 + 水平扩展

## 快速诊断指南

若生产环境出现异常，请按以下顺序排查：

1. **用户反馈报错？** 检查健康检查端点，随后查看日志中的错误率，最后确认 LLM 提供商的状态页面
2. **响应缓慢？** 检查 P99 延迟，随后查看缓存命中率，最后检查追踪链路中的 LLM 响应时间
3. **成本激增？** 检查单次请求成本趋势，随后查看缓存命中率，接着按成本排序查看头部用户，最后排查是否因提示词模板变更导致 Token 数量增加
4. **质量下降？** 检查是否部署了新版提示词，检查检索增强生成 (Retrieval-Augmented Generation, RAG) 的检索准确率是否变化，检查模型提供商是否更改了默认模型版本
5. **安全事件？** 检查护栏拦截率（骤降通常意味着护栏被禁用），检查请求日志中是否存在异常模式，并立即轮换 API 密钥
