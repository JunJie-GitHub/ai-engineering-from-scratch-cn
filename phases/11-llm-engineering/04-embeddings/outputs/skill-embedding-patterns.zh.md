# 嵌入向量（Embedding）模式

每个嵌入向量工作流都遵循以下契约：

text -> embed(text) -> vector (float array)
similarity(vector_a, vector_b) -> score (float)

嵌入模型（Embedding Model）与相似度度量（Similarity Metric）是唯一需要重点考量的两个决策，其余部分皆为底层基础设施的搭建。

## 何时使用嵌入向量

- 跨文档语义搜索（Semantic Search）（关注语义而非关键词匹配）
- 相似项聚类（如客服工单、产品评论、缺陷报告）
- 基于最近邻（Nearest Neighbors）的分类（通过与已标注样本的相似度为新数据打标签）
- 推荐系统（寻找与用户偏好相似的物品）
- 去重处理（利用相似度阈值识别近似重复内容）

## 何时不应使用嵌入向量

- 精确关键词匹配（应使用全文检索 Full-Text Search）
- 结构化查询（应使用 SQL 或过滤器）
- 数据量较小且人工标注更快的场景（少于 100 条数据）
- 可解释性（Explainability）优先于准确率的场景（嵌入向量具有黑盒特性）

## 模型选择

请根据实际约束条件进行选择：

- **需要 API 且追求性价比**：OpenAI text-embedding-3-small（1536 维，0.02 美元/百万 Token）
- **追求最高准确率**：Voyage-3（1024 维，0.06 美元/百万 Token，MTEB 榜单得分最高）
- **需要本地部署/私有化**：BGE-M3（1024 维，免费，支持多语言，推荐使用 GPU）
- **需要快速本地原型开发**：all-MiniLM-L6-v2（384 维，免费，可在 CPU 上运行）
- **需要多语言支持**：Cohere embed-v3（1024 维）或 BGE-M3（两者均具备强大的多语言能力）

核心原则：索引（Indexing）与查询（Querying）阶段绝不可混用不同的嵌入模型。不同模型生成的向量存在于互不兼容的向量空间中。

## 文本分块（Chunking）规则

1. 每个分块目标长度为 256-512 个 Token，并设置 50 个 Token 的重叠（Overlap）
2. 尽量避免在句子中间截断
3. 为每个分块附带元数据（Metadata）（如源文件、章节标题、位置信息）
4. 针对结构化文档（如 Markdown、HTML），优先在标题边界处进行切分
5. 通过检索已知答案并检查召回结果来验证分块质量

## 相似度度量选择

- **余弦相似度（Cosine Similarity）**：默认选项，适用于变长文本，已进行归一化处理
- **点积（Dot Product）**：当向量已进行单位归一化时使用（OpenAI 模型默认已处理），计算速度略快
- **欧氏距离（Euclidean Distance）**：适用于聚类任务，或当向量的绝对位置具有重要意义时

当向量经过归一化处理后，这三种度量方式给出的排序结果完全一致。该选择仅对未归一化的向量产生影响。

## 存储优化

提供三个可叠加的压缩层级：

1. **套娃截断（Matryoshka Truncation）**：降低维度（1536 维降至 256 维，节省 6 倍空间，准确率损失 3-5%）
2. **Float16 量化（Quantization）**：每维度存储空间减半（节省 2 倍空间，准确率损失 <1%）
3. **二值量化（Binary Quantization）**：每维度仅占 1 比特（节省 32 倍空间，准确率损失 5-10%，需配合重排序使用）

生产环境最佳实践：在全量语料库上使用二值向量进行初筛召回，随后使用 Float32 向量对 Top-1000 结果进行重打分（Rescoring）。

## 检索后重排序（Retrieve-then-Rerank）

追求最高准确率的两阶段流水线：

1. 双编码器（Bi-encoder）召回 Top-100 候选结果（速度快，依赖预计算的嵌入向量）
2. 交叉编码器（Cross-encoder）重排序至 Top-10（速度较慢，需逐一处理查询-文档对）

该方案在精确率（Precision）指标上比单阶段检索高出 10-15%。适用于准确率优先级高于延迟（Latency）的场景。

## 常见误区

- 索引与查询阶段使用不同的嵌入模型
- 对整篇文档而非分块进行向量化（导致向量沦为所有内容的平均值，丢失细节）
- 计算余弦相似度前未对向量进行归一化（多数模型已内置预处理，但仍需核实）
- 忽略分块重叠设置（边界处的句子会丢失上下文信息）
- 仅存储向量而丢弃原始文本（检索与展示均需原始文本）
- 模型更新后未重新生成向量（旧向量与新模型不兼容）
- 仅凭准确率选择向量维度（存储成本与延迟会随维度线性增长）

## 嵌入向量调试

若搜索结果不佳：

1. 验证查询嵌入向量（query embedding）是否非零（空输入或仅含空格的输入会生成零向量）
2. 手动检查已知相关文档的相似度得分（similarity score）
3. 尝试改写查询语句，使其与文档词汇相匹配
4. 检查文本分块（chunk）边界，确保相关内容未被拆分到不同的块中
5. 对比不同相似度度量指标（余弦相似度 cosine、点积 dot、欧氏距离 euclidean）下的 Top-k（top-k）结果，以排查归一化（normalization）问题
6. 使用完全匹配的查询（直接复制文档中的一句话）进行测试，以确认整个处理流水线（pipeline）运行正常

## 生产环境参数

- 分块大小（chunk size）：256-512 个 token
- 分块重叠（chunk overlap）：50 个 token（占分块大小的 10-20%）
- Top-k 检索（top-k retrieval）：直接使用时取 5-10，用于重排序（reranking）时取 50-100
- 相似度阈值（similarity threshold）：余弦相似度需达到 0.7 以上（低于此值的结果通常不相关）
- 批量嵌入（batch embedding）：每次 API 调用处理 100-500 条文本以提升吞吐量
- 索引重建（index rebuild）：当模型变更或文档发生显著更新时，需重新生成嵌入向量
