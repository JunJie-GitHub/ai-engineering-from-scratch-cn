# 检索增强生成（RAG）流水线模式

每个 RAG 系统都遵循以下模式：

documents -> chunk -> embed -> store
query -> embed -> search(top_k) -> build_prompt -> generate

索引（Indexing）针对每个文档仅执行一次。查询（Querying）则在每次用户请求时触发。

## 何时使用 RAG

- 大语言模型（LLM）需要访问私有或最新文档
- 微调（Fine-tuning）成本过高或更新速度太慢
- 需要为答案提供来源引用
- 知识库频繁更新

## 何时不应使用 RAG

- 答案属于大语言模型已掌握的通用知识
- 任务属于创意类（如写作、头脑风暴）而非事实类
- 需要模型采用特定的推理风格（此时应使用微调）

## 实施清单

1. 将文档切分为 256-512 个词元（Token）的文本块（Chunk），并设置 50 个词元的重叠
2. 使用统一的嵌入模型（Embedding Model）对每个文本块进行向量化
3. 将向量与原始文本一同存储至向量数据库（Vector Database）
4. 查询时，使用相同模型对用户问题进行向量化
5. 通过余弦相似度（Cosine Similarity）检索最相似的前 k（Top-k，5-10）个文本块
6. 构建提示词（Prompt）：系统指令 + 检索到的上下文 + 用户问题
7. 生成答案，并确保其严格基于检索到的上下文
8. 返回答案并附带来源引用

## 常见错误

- 索引和查询阶段使用不同的嵌入模型（导致向量空间不兼容）
- 文本块过小（丢失上下文）或过大（稀释相关性）
- 文本块间未设置重叠（导致在边界处切断句子）
- 文档更新后忘记重新建立索引
- 直接将检索到的文本块返回给用户，而未生成连贯的答案
- 事实类 RAG 查询未设置 temperature=0（温度值越高，幻觉现象越严重）

## 检索调试

如果未能检索到正确的文本块：
1. 打印查询向量，并验证其非全零
2. 手动检查已知相关文本块的余弦相似度
3. 尝试改写查询语句，使其与文档词汇相匹配
4. 确认索引阶段与查询阶段使用的嵌入模型一致
5. 检查相关内容是否在文档切分过程中丢失

## 生产环境参数

- 文本块大小：256-512 个词元
- 重叠长度：50 个词元（约占文本块大小的 10-20%）
- Top-k：适用于大多数场景的 5-10 个
- temperature：事实类答案设为 0
- 嵌入模型：text-embedding-3-small（性价比高）或 text-embedding-3-large（精度更高）
