# 大语言模型（LLM）成本优化模式

在构建需要控制成本的大语言模型应用时，请应用此决策框架。

## 何时进行优化

**立即进行优化的情况：**
- 每月 LLM 支出超过 500 美元或占基础设施预算的 10%
- 面向消费者的产品单次查询成本超过 0.01 美元
- 系统提示词（System Prompt）超过 1,000 个词元（token）且随每次请求发送
- 超过 30% 的查询为重复或高度相似查询
- 日活跃用户（DAU）正从 100 扩展至 10,000 以上

**暂不优化的情况：**
- 日活跃用户不足 100 且仍在验证产品市场契合度（Product-Market Fit）
- 每月支出低于 100 美元且增长缓慢
- 仍在迭代提示词设计（缓存会导致提示词固化）

## 缓存策略选择

### 精确缓存（Exact Caching）

**适用场景：** temperature=0、提示词完全重复、需要确定性输出。

key = sha256(json.dumps({"model": m, "messages": msgs, "temp": 0}))

- 实现耗时：30 分钟
- 命中率：大多数应用为 10-25%，常见问题解答（FAQ）机器人可达 40-60%
- 延迟：<1 毫秒（字典查找）
- 风险：若底层数据发生变更，可能返回过时响应

**跳过场景：** temperature > 0、每次查询均唯一、需要实时数据。

### 语义缓存（Semantic Caching）

**适用场景：** 用户用不同措辞询问相同问题、FAQ 密集型产品、客户支持场景。

- 实现耗时：2-4 小时（嵌入向量（Embedding）+ 相似度计算 + 存储）
- 命中率：在精确缓存基础上额外提升 15-35%
- 延迟：10-50 毫秒（嵌入向量 + 近似最近邻搜索（ANN Search））
- 风险：误报（为相似但不同的问题返回错误的缓存答案）

**阈值指南：**
- 0.98+：非常保守，几乎无误报，命中率较低
- 0.95：事实性问答的良好平衡点
- 0.90：较为激进，命中率更高但存在答错风险
- 0.85：仅适用于非关键应用（如建议、自动补全）

**跳过场景：** 每次查询上下文均唯一（如代码生成）、响应必须反映最新数据、查询空间无边界。

### 供应商提示词缓存（Provider Prompt Caching）

**适用场景：** 系统提示词超过 1,024 个 token（OpenAI）或达到模型特定最低要求，且重复发送相同前缀。

| 供应商 | 操作 | 节省成本 |
|----------|--------|---------|
| Anthropic | 在系统消息中添加 `cache_control: {"type": "ephemeral"}` | 缓存前缀节省 90%（扣除 25% 写入溢价后） |
| OpenAI | 无需操作（自动生效） | 缓存前缀节省 50% |
| Google | 使用带明确生存时间（TTL）的上下文缓存 API | 缓存上下文节省约 75% |

**跳过场景：** 系统提示词随每次请求变化、提示词长度低于最低要求。

## 模型路由规则

### 基于关键词（简单、快速）

simple:  <= 5 words OR matches FAQ keywords -> gpt-4o-mini ($0.15/$0.60)
medium:  general queries, summaries        -> claude-sonnet ($3/$15)
complex: "analyze", "compare", "debug"     -> gpt-4o ($2.50/$10)

- 实现耗时：1 小时
- 准确率：70-80%
- 节省成本：模型成本的 40-60%

### 基于嵌入向量（更准确）

为每个类别嵌入 50-100 条已标注查询。通过最近邻算法对新查询进行分类。

- 实现耗时：4-8 小时
- 准确率：85-92%
- 节省成本：模型成本的 50-70%
- 额外成本：分类嵌入向量约 0.02 美元/百万 token（可忽略不计）

### 基于机器学习（生产级）

基于历史查询/模型配对数据训练小型分类器（逻辑回归或小型 BERT）。

- 实现耗时：1-2 周
- 准确率：90-95%
- 节省成本：模型成本的 60-75%
- 要求：来自生产流量的已标注训练数据

## 速率限制配置

### 按层级划分的令牌桶（Token Bucket）参数

| 层级 | 桶容量 | 填充速率 | 最大每分钟请求数（RPM） | 每日上限 |
|------|-------------|-------------|---------|-----------|
| 免费版 | 5 万 token | 500/秒 | 10 | 5 万 |
| 专业版 | 50 万 token | 5000/秒 | 60 | 50 万 |
| 企业版 | 500 万 token | 5 万/秒 | 300 | 500 万 |

### 实施检查清单

1. 在多实例应用中，将限流桶（Buckets）存储在 Redis 中（而非内存中）
2. 使用原子操作（Atomic Operations）（MULTI/EXEC）以防止竞态条件（Race Conditions）
3. 在拒绝响应中返回 `Retry-After` 请求头
4. 将被拒绝的请求作为指标进行追踪（拒绝率 >5% 表示层级限制设置过严）
5. 实现优雅降级（Graceful Degradation）：优先拒绝高成本模型请求，保留低成本模型的访问权限

## 预算控制

### 三阈值熔断器（Circuit Breaker）

| 阈值 | 操作 | 可逆 |
|-----------|--------|------------|
| 月度预算的 70% | 记录警告，通过 Slack/PagerDuty 通知团队 | 是（自动） |
| 月度预算的 85% | 将所有流量路由至最便宜的模型 | 是（自动，下个计费周期） |
| 月度预算的 95% | 仅提供缓存响应，拒绝新的大语言模型（LLM）调用 | 是（手动重置或下个周期） |

### 单用户成本追踪

追踪每位用户的累计成本。标记超出中位数 10 倍的用户。常见原因包括：
- 合法的高频用户（升级其服务层级）
- 提示词注入循环（Prompt Injection Loop）（机器人发送自动化请求）
- 低效的集成方式（客户端在每次报错时都进行重试）

### 需展示在仪表盘上的关键指标

- **单次查询成本**（P50、P95、P99）——按模型、按功能、按用户层级划分
- **缓存命中率**（Cache Hit Rate）——精确匹配与语义匹配，随时间变化的趋势
- **模型分布**——各模型流量占比、各模型成本
- **预算消耗速率**（Budget Burn Rate）——当前支出与按当前速率推算的月度预算对比
- **拒绝率**——受速率限制（Rate-Limited）的请求百分比，按层级划分

## 常见错误

| 错误做法 | 负面影响 | 解决方案 |
|---------|-------------|-----|
| 对温度参数（Temperature）> 0 的请求进行缓存 | 输出具有非确定性，过期缓存会导致结果多样性错误 | 仅缓存 temperature=0 的调用，或接受缓存响应会丧失随机性 |
| 语义缓存阈值设置过低 | 对表面相似的查询返回错误答案 | 初始值设为 0.95，仅在测量误报率（False Positive Rate）后再酌情调低 |
| 未设置缓存失效机制 | 底层数据变更时，响应内容会过期 | 设置生存时间（TTL）（动态数据 1 小时，静态数据 24 小时），数据更新时主动失效缓存 |
| 将所有流量路由至最便宜的模型 | 质量下降，用户会察觉 | 按复杂度进行路由，按层级评估质量，设置最低质量阈值 |
| 未设置单用户限制 | 单个滥用用户即可耗尽全部预算 | 始终实施单用户配额，即使额度设置得较为宽松 |
| 忽略输出 Token 成本 | 每个输出 Token 的成本是输入 Token 的 2-5 倍 | 合理设置 `max_tokens`，使用停止序列（Stop Sequences），压缩输出内容 |
| 在提示词稳定前启用缓存 | 缓存会被旧提示词的响应填满 | 仅在提示词定稿后启用缓存，提示词变更时清空缓存 |
