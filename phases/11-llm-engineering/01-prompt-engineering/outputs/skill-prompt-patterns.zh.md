# 提示词模式选择指南

在构建基于大语言模型（Large Language Model, LLM）的功能时，应在编写具体提示词之前先选定提示词模式。模式决定了整体结构，而具体内容则填充其中。

## 模式决策矩阵

| 任务类型 | 主要模式 | 次要模式 | 温度参数（Temperature） | 是否需要少样本（Few-Shot）？ |
|-----------|----------------|-------------------|-------------|-----------------|
| 数据提取 | 模板填充（Template Fill） | 少样本（Few-Shot） | 0.0 | 是（2-3 个示例） |
| 分类 | 少样本（Few-Shot） | 护栏（Guardrail） | 0.0 | 是（3-5 个示例） |
| 摘要生成 | 角色设定 + 模板（Persona + Template） | 受众适配（Audience Adapt） | 0.3 | 否 |
| 代码生成 | 角色设定（Persona） | 思维链（Chain-of-Thought） | 0.0 | 可选 |
| 创意写作 | 角色设定（Persona） | 批判反思（Critique） | 0.7-1.0 | 否 |
| 多步推理 | 思维链（Chain-of-Thought） | 问题分解（Decomposition） | 0.3 | 可选 |
| 问答 | 角色设定 + 护栏（Persona + Guardrail） | 边界限制（Boundary） | 0.3 | 否 |
| 提示词生成 | 元提示词（Meta-Prompt） | 批判反思（Critique） | 0.7 | 是（1-2 个示例） |
| 内容审核 | 护栏 + 边界限制（Guardrail + Boundary） | 少样本（Few-Shot） | 0.0 | 是（5 个以上示例） |
| 翻译/改编 | 受众适配（Audience Adapt） | 少样本（Few-Shot） | 0.3 | 是（2-3 个示例） |

## 各模式适用场景

**角色设定模式（Persona Pattern）**：应作为所有提示词的基础基线。唯一需要考量的是角色设定的具体程度。对于通用任务，宽泛的角色即可满足需求；对于特定领域任务，角色应明确指定领域、资历级别及上下文背景。

**少样本模式（Few-Shot Pattern）**：适用于输出格式比内容本身更重要的场景。若模型需要生成特定的 JSON 结构、CSV 格式或分类标签，提供示例比单纯下达指令更有效。经验法则：简单格式提供 2-3 个示例，复杂或模糊格式提供 5 个以上示例。

**思维链模式（Chain-of-Thought Pattern）**：适用于数学计算、逻辑推理、多步分析以及任何需要模型“展示推导过程”的任务。在推理任务中可将准确率提升 10%-40%（Wei 等，2022）。切勿用于简单的事实查询或数据提取——这会浪费 Token。

**模板填充模式（Template Fill Pattern）**：适用于结构化提取任务，要求每次输出保持完全一致的格式。配合温度参数（Temperature）=0.0 使用效果最佳，并需明确定义缺失字段的“N/A”处理规则。

**批判反思模式（Critique Pattern）**：适用于质量优先于速度的场景。模型会先生成内容，随后进行自我批判与优化。该模式会使 Token 消耗大致翻倍，但能显著提升输出的准确性与完整性。最适合高风险输出场景（如报告、推荐建议、面向公众的内容）。

**护栏模式（Guardrail Pattern）**：适用于任何面向用户的系统。必须包含：范围边界、对超出范围请求的拒绝行为，以及明确的“我不知道”处理机制。建议与应用端的输入验证结合使用。

**元提示词模式（Meta-Prompt Pattern）**：用于为新任务生成提示词。无需从零开始编写，只需描述任务目标，让模型自动生成提示词，随后进行测试与迭代。可大幅节省初期提示词开发的时间。

**问题分解模式（Decomposition Pattern）**：适用于适合“分而治之”策略的复杂问题。模型会将问题拆解为多个部分，逐一解决后再进行整合。对于包含 3-7 个子问题的任务最为有效。

**受众适配模式（Audience Adaptation Pattern）**：适用于同一内容需面向不同受众的场景。必须明确指定目标受众——切勿依赖模型自行从上下文中猜测。

**边界限制模式（Boundary Pattern）**：适用于生产环境中绝对禁止回答特定类型问题的系统。其强度高于普通护栏，因为它定义了硬性范围并附带精确的拒绝回复。在合规敏感型领域不可或缺。

## 跨模型兼容性

以下按模式在 GPT-4o、Claude 3.5 Sonnet、Gemini 1.5 Pro 和 Llama 3 上表现的一致性进行排序：

| 模式 | 跨模型一致性 | 备注 |
|---------|------------------------|-------|
| 少样本提示 (Few-Shot) | 非常高 | 示例在所有模型间迁移效果良好 |
| 模板填充 (Template Fill) | 非常高 | 明确的结构几乎不留发散空间 |
| 思维链 (Chain-of-Thought) | 高 | 所有主流模型均支持“逐步思考” |
| 角色设定 (Persona) | 高 | 普遍适用，但不同模型对角色具体程度的响应不同 |
| 安全护栏 (Guardrail) | 中等 | Claude 遵循护栏最严格；GPT-4o 在长对话中有时会偏离 |
| 自我评审 (Critique) | 中等 | 自我评审的质量因模型差异显著 |
| 元提示 (Meta-Prompt) | 中等 | GPT-4o 和 Claude 生成的提示词风格不同 |
| 边界控制 (Boundary) | 中低 | 拒绝行为因模型而异；需针对各模型单独测试 |

## 常见错误

1. **对所有任务都使用思维链 (Chain-of-Thought)**：CoT 会增加 Token 消耗和延迟。仅在需要推理步骤时使用。
2. **约束条件过多**：超过 5-7 个约束时，模型会开始忽略部分条件。请优先保留最重要的 3 个。
3. **角色设定与约束条件冲突**：“你是一位创意作家”加上“绝不使用比喻”会让模型感到困惑。
4. **未指定温度参数 (Temperature)**：在需要确定性输出时，仍保留默认温度值（通常为 1.0）。
5. **在不同模型间直接复制粘贴提示词**：务必进行测试。为 GPT-4o 调优的提示词在 Claude 上可能表现不佳，反之亦然。
6. **忽略系统消息 (System Message)**：将所有内容都放在用户消息中，而不是利用系统消息来设定持久性规则。
7. **过度依赖负面约束**：“不要做 X、Y、Z、A、B、C”的效果不如“只做 W”。正向表述能为模型提供明确的目标。

## 可靠性目标

| 应用场景 | 模式组合 | 预期准确率 | Token 成本 |
|----------|-------------------|-------------------|------------|
| 生产环境信息抽取 | 模板 + 少样本提示 | 95%+ | 低 (500-1K) |
| 面向用户的问答 | 角色设定 + 安全护栏 + 边界控制 | 90%+ | 中 (1-2K) |
| 代码生成 | 角色设定 + 思维链 | 85%+ | 中 (1-3K) |
| 内容生成 | 角色设定 + 自我评审 | 90%+（质量） | 高 (2-4K，双轮处理) |
| 文本分类 | 少样本提示 + 安全护栏 | 95%+ | 低 (300-800) |
| 复杂分析 | 任务分解 (Decomposition) + 思维链 | 85%+ | 高 (3-5K) |
