# 奖励模型设计器（Reward Model Designer）

在构建 RLHF（Reinforcement Learning from Human Feedback，基于人类反馈的强化学习）流水线以将语言模型对齐至目标行为（如帮助性、编程能力、安全性、诚实性）时，请使用本框架来设计数据收集协议、训练奖励模型（Reward Model）并配置 PPO（Proximal Policy Optimization，近端策略优化）。

## 输入要求

请提供：
- **目标行为**（例如：“有益且无害的助手”、“Python 编程专家”、“具备安全性的医疗问答”）
- **基础模型**（例如：经过 SFT（Supervised Fine-Tuning，监督微调）的 Llama 3 8B、Mistral 7B Chat）
- **奖励模型规模**（通常与策略模型（Policy Model）相同或更大）
- **标注预算**（可用的人工工时或对比样本对数量）
- **算力预算**（用于奖励模型训练与 PPO 的 GPU 工时）

## 步骤 1：偏好数据收集

### 标注协议

1. **提示词选择**：从 SFT 训练分布中采样，并加入分布外提示词（10-20% 为新颖/未见过的提示词）
2. **回复生成**：使用 SFT 模型，设置不同的温度参数（Temperature）（0.3、0.7、1.0），为每个提示词生成 2-4 条回复
3. **对比格式**：向标注人员展示恰好 2 条回复，并提问“哪条回复更好？”
4. **评估标准**：针对您的具体应用场景，明确定义“更好”的含义

### 评估标准模板

| 评估维度 | 权重 | 描述 |
|-----------|--------|-------------|
| 帮助性 | 40% | 是否完整且正确地回答了问题？ |
| 无害性 | 25% | 是否避免了有害、偏见或误导性内容？ |
| 诚实性 | 20% | 是否承认不确定性而非产生幻觉（Hallucination）？ |
| 简洁性 | 15% | 回复长度是否与该问题相匹配？ |

请根据您的具体应用场景调整权重。例如，编程助手可能将正确性权重设为 60%，简洁性设为 20%。

### 数据规模指南

| 规模级别 | 对比样本对数量 | 标注人工时 | 预期奖励模型准确率 |
|-------|-----------------|-----------------|---------------------|
| 最小可行规模 | 5,000-10,000 | 400-800 | 60-65% |
| 生产环境 v1 | 20,000-50,000 | 1,600-4,000 | 65-72% |
| 生产环境 v2 | 100,000-500,000 | 8,000-40,000 | 72-78% |

InstructGPT 使用了来自 40 名承包商的 33,000 组对比数据。Anthropic 的初始论文使用了来自 20 名标注人员的 22,000 组数据。标注人员间一致性（Inter-annotator agreement）通常为 70-75%——奖励模型的性能上限无法超越人类的一致性水平。

### 质量控制

- **一致性过滤**：剔除标注人员一致同意率低于 70% 的样本对
- **标注人员校准**：在正式标注前，使用已知的高质量样本对进行校准轮次测试
- **偏差检测**：监控标注人员是否持续偏好更长的回复、更正式的语言或特定模式
- **对抗性样本**：包含 5-10% 专门设计的样本，用于识别未仔细阅读的标注人员

## 步骤 2：奖励模型架构

### 架构决策

| 决策项 | 建议方案 | 依据 |
|----------|---------------|-----------|
| 基础架构 | 与策略模型使用相同的 Transformer | 从 SFT 检查点（Checkpoint）初始化权重可提供强大的初始特征 |
| 输出头（Output Head） | 从最后一个隐藏状态进行单次线性投影 | 从最完整的位置表示中提取标量奖励值 |
| 模型规模 | ≥ 策略模型规模 | 规模较小的奖励模型会产生不可靠的信号，从而导致 PPO 训练不稳定 |
| 初始化方式 | 使用 SFT 检查点并替换新的输出头 | 预训练特征已能充分捕捉语言质量 |

### 训练配置

| 参数 | 范围 | 说明 |
|-----------|-------|-------|
| 学习率 (Learning rate) | 1e-5 至 5e-5 | 低于监督微调 (Supervised Fine-Tuning, SFT)，因为该任务相对更简单 |
| 训练轮数 (Epochs) | 1-3 | 比较数据有限时，过拟合是主要风险 |
| 批次大小 (Batch size) | 64-256 | 每个“样本”为一对数据，因此有效数据量翻倍 |
| 损失函数 (Loss function) | Bradley-Terry: `-log(sigmoid(r_preferred - r_rejected))` | 成对比较的标准做法 |
| 验证集划分比例 (Validation split) | 10-20% | 在预留的配对数据上监控准确率 |

### 评估指标 (Evaluation Metrics)

1. **成对准确率 (Pairwise accuracy)**：奖励模型 (Reward Model, RM) 在预留的偏好配对中正确排序的比例是多少？目标：> 65%
2. **差值分布 (Margin distribution)**：绘制 `(r_preferred - r_rejected)` 的分布图。分布中心应位于 0 以上，且负值极少。
3. **校准度 (Calibration)**：`sigmoid(r_preferred - r_rejected)` 的值是否接近人类实际的偏好概率？
4. **分布外泛化能力 (Out-of-Distribution, OOD generalization)**：使用与训练集分布不同的提示词 (prompts) 进行测试。准确率下降幅度应 < 10%。

## 步骤 3：近端策略优化 (Proximal Policy Optimization, PPO) 配置

### 超参数 (Hyperparameters)

| 参数 | 典型值 | 设置过高的影响 | 设置过低的影响 |
|-----------|--------------|-------------------------|------------------------|
| KL 散度系数 (KL coefficient, beta) | 0.01-0.05 | 模型几乎无法学习，输出过于接近 SFT 模型 | 奖励黑客攻击 (Reward hacking)，输出退化 |
| 学习率 (Learning rate) | 5e-6 至 3e-5 | 训练不稳定，模型发散 | 收敛缓慢，浪费算力 |
| 裁剪比例 (Clip ratio, epsilon) | 0.1-0.3 | 更新步幅过大，可能导致不稳定 | 更新过于保守，学习缓慢 |
| 每批次 PPO 训练轮数 (PPO epochs per batch) | 1-4 | 对当前批次过拟合 | 未充分利用每个批次的数据 |
| 生成批次大小 (Generation batch size) | 128-512 | 内存不足 | 梯度估计噪声过大 |
| 最大响应长度 (Max response length) | 256-1024 | 生成速度慢，内存占用高 | 截断有用的回复内容 |

### 监控面板 (Monitoring Dashboard)

在 PPO 训练期间跟踪以下指标：

1. **平均奖励 (Mean reward)**：训练过程中应呈上升趋势。达到平台期属于正常；若下降则表明训练不稳定。
2. **KL 散度 (KL divergence)**：应保持在 10-20 nats 以下。出现尖峰通常意味着奖励黑客攻击。
3. **响应长度 (Response length)**：应保持稳定。若单调递增，则表明模型通过增加冗余内容来骗取奖励。
4. **熵 (Entropy)**：Token 分布的熵应缓慢下降。快速下降意味着模式崩溃 (Mode collapse)。
5. **奖励模型一致性 (Reward model agreement)**：使用奖励模型对 PPO 生成的回复进行打分，一致性应逐步提升。

### PPO 训练中的危险信号 (Red Flags During PPO)

| 现象 | 可能原因 | 解决方案 |
|---------|-------------|-----|
| 奖励上升但输出质量下降 | 奖励黑客攻击 (Reward hacking) | 提高 KL 散度系数，使用对抗样本重新训练奖励模型 (RM) |
| KL 散度急剧上升 | 学习率过高或 KL 散度系数过低 | 降低学习率，提高 beta 值 |
| 响应长度单调递增 | 奖励模型偏好冗长回复 | 在奖励函数中添加长度惩罚项，使用长度受控的配对数据重新训练 RM |
| 所有回复变得完全相同 | 模式崩溃 (Mode collapse) | 提高生成温度 (temperature)，减少 PPO 训练轮数 |
| 奖励剧烈震荡 | PPO 训练不稳定 | 降低学习率，提高裁剪比例 (clip ratio) |

## 步骤 4：端到端验证 (End-to-End Validation)

在部署经过基于人类反馈的强化学习 (Reinforcement Learning from Human Feedback, RLHF) 训练的模型之前：

1. **与 SFT 模型进行 A/B 测试**：在 200 个以上的测试提示词上分别运行 SFT 和 RLHF 模型。安排 3 名以上的评估人员对比回复。RLHF 模型的胜率应 > 60%。
2. **安全性评估**：使用已知的对抗性提示词（如越狱攻击、有害请求）进行测试。RLHF 模型应能做出恰当的拒绝。
3. **回归检查**：运行标准基准测试（如 MMLU、HumanEval、MT-Bench），以确认 RLHF 模型未丧失核心能力。
4. **遗忘检查**：在通用文本语料库上测量困惑度 (Perplexity)。相较于 SFT 模型，其增幅应 < 10%。
5. **长度分析**：对比 SFT 与 RLHF 模型的平均响应长度。若 RLHF 模型回复长度超出 50% 以上，则奖励模型很可能存在冗长偏好偏差。
