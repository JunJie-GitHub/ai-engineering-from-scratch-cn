# 投机解码（Speculative Decoding）策略选择器

帮助工程师在基础投机解码（Vanilla）、Medusa、EAGLE 或前瞻解码（Lookahead）之间进行选择，并针对特定工作负载调整 `N`（草稿长度）。

## 需收集的信息

1. **验证模型（Verifier Model）** — 负责生成最终输出的 LLM。模型规模至关重要（草稿模型的推理成本必须低于验证模型才能实现加速）。
2. **工作负载类型** — 代码生成、对话、结构化输出、摘要。该类型决定了接受率（Acceptance Rate）。
3. **采样策略** — 贪婪采样、低温度（low-T）、高温度（high-T）、束搜索（beam）。高温采样会显著降低接受率。
4. **硬件目标** — 内存预算决定了是否能够部署独立的草稿模型（Draft Model）。
5. **工程预算** — Medusa 和 EAGLE 需要微调；vanilla 和 lookahead 则不需要。
6. **延迟目标** — 交互式对话（首词延迟 TTFT <500ms，单 token 延迟 <50ms）与批处理（吞吐量优先）。

## 决策规则

- **快速启动，无需训练**：使用同系列 1B–3B 模型的 vanilla 草稿方案。通常可实现 2 倍加速。
- **具备微调能力**：使用验证模型的隐藏状态（Hidden States）训练 EAGLE-2 或 EAGLE-3。通常可实现 3–4 倍加速。
- **可微调但无法运行双模型**：采用 Medusa（在验证模型上增加额外输出头）。通常可实现 2–3 倍加速。
- **无训练预算且无可用草稿模型**：采用前瞻解码（Lookahead Decoding）。通常可实现 1.3–1.6 倍加速。
- **高并发批处理服务**：连续批处理（Continuous Batching）更为关键；随着批次增大，验证模型已趋于饱和，投机解码的收益会逐渐递减。
- **高温或随机采样**：接受率会急剧下降。建议降低 `N` 值（2–3）或直接禁用该功能。
- **结构化输出（JSON、代码）**：接受率较高。可将 `N` 提升至 7 以上以获取最大加速比。

## 参数调优

- **N（草稿长度）**：初始值设为 5。测量接受率。若 α > 0.9，可提升至 7；若 α < 0.6，则降至 3。
- **草稿温度（Draft Temperature）**：需与验证模型的温度保持一致。采样温度不匹配会导致 α 下降。
- **树深度（EAGLE-2 / Medusa）**：3–5 个分支；仅在 α > 0.8 时，更宽的树结构才能带来收益。
- **草稿模型规模**：选择能达到 α > 0.7 的最小模型。为 70B 验证模型搭配 1B 草稿模型是常见做法；切勿低于验证模型的分词器（Tokenizer）/ 词嵌入（Embedding）兼容性要求。

## 关键提醒

- 确保草稿模型与验证模型使用相同的分词器。不同的 BPE 切分规则会破坏投机解码的加速保证。
- 投机解码与 vLLM 中的连续批处理存在交互影响：当批次已饱和时，单请求的加速比会下降。
- EAGLE 的隐藏状态输入需要访问验证模型的内部结构；Hugging Face API 未必总是暴露这些接口。建议优先使用 vLLM 或 SGLang 运行时。
- Medusa 的额外输出头需要基于验证模型自身的输出进行监督微调（Supervised Fine-Tuning）。数据收集步骤通常是主要的成本开销。

## 输出格式

返回内容应包含：

1. **推荐方案** — 指定一种策略名称及调优参数（例如 "EAGLE-2, N=5, tree_depth=4"）。
2. **预期加速比** — 需明确给出 α 的假设值。
3. **兼容性检查** — 分词器匹配情况、运行时支持、KV 缓存回滚（KV Cache Rollback）支持。
4. **降级方案** — 若主策略表现不佳，下一步应尝试的备选方案。
5. **评估方案** — 如何在代表性样本上验证接受率与加速比。
