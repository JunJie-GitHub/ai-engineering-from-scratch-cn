根据模型大小、服务等级协议 (SLA，首字元延迟 (TTFT) 的 P99 分位数)、流量特征（平稳型与突发型）以及预算策略，制定一份冷启动 (cold-start) 缓解计划。

输出内容：

1. 冷启动预算。拆解原始冷启动路径（节点配置、镜像拉取、权重加载至高带宽内存 (HBM)、引擎初始化、首次前向传播）。针对指定模型大小，采用 2026 年基准秒数进行估算。
2. 层级选择。选择能将总耗时降至 SLA 以下的最少层级组合：预置镜像 (pre-seeded image, L1)、模型流式加载 (model streamer, L2)、GPU 快照 (GPU snapshot, L3)、预热池 (warm pool, L4)、分层加载 (tiered loading, L5)。针对每个层级所针对的具体阶段进行合理性论证。
3. 预热池规模设定。明确主路径的 `min_workers` 参数。若 70B 及以上模型的 SLA 要求 TTFT P99 < 60 秒，则无论成本如何，必须强制启用预热池。
4. 成本估算。所选预热池的月度 GPU 成本，以及每日预期的冷启动次数。
5. 尾部请求策略。全新副本上的首位用户将如何处理——是排队等待预热副本，还是必须承担冷启动开销？请明确具体策略（例如：“10 秒内将首个请求路由至任意预热副本；超时则降级至冷启动”）。
6. 故障模式。若预热副本在会话期间宕机将如何处理。恢复是自动进行的（如实时迁移 (live migration)），还是在下一次请求时触发冷启动？

硬性拒绝条件：
- 未计算月度成本便直接提议“只需添加预热池”。
- 提出缓解策略却未指明其针对的具体阶段（例如：“使用 Bottlerocket”却未说明其可消除 180 秒的镜像拉取时间）。
- 忽略 GPU 快照的每 GPU 拓扑限制——若平台迁移实例规格 (SKU)，快照将失效。

拒绝规则：
- 若 SLA 要求全新 70B 模型冷启动的 TTFT P99 < 5 秒且无预热池，则直接拒绝——以 2026 年的基础设施速度在数学上不可行。
- 若预算不允许使用预热池，但 SLA 要求冷启动时间低于 30 秒，需指明特定平台的解决方案（如 Modal GPU 快照、Baseten 预热机制），并拒绝在未采用该方案的其他平台上承诺此 SLA。
- 若运维人员要求针对突发流量和 70B 模型实现缩容至零 (scale-to-zero)，则拒绝承诺 SLA——在缺乏快照或预热池的情况下，该数学模型无法成立。

输出要求：一份单页计划，列出各阶段、层级、`min_workers`、月度成本、尾部请求策略及故障模式。结尾需注明唯一告警指标：过去滚动一小时内的 P99 冷启动耗时。
