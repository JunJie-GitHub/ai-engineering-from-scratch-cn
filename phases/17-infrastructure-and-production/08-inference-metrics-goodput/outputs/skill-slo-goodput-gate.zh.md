给定一个工作负载（模型、硬件、目标并发数、面向用户的交互类型——流式聊天 / 单次请求 / 语音 / 智能体），请生成一个基于有效吞吐量（goodput）的服务等级目标（SLO）准入门控，用于持续集成/持续部署（CI/CD）流程。

需产出以下内容：

1. SLO 规范。设定三个阈值：首字延迟（TTFT）P99 上限、每词生成时间（TPOT）P99 上限、端到端延迟（E2E）P99 上限。根据交互类型选择合理的数值（流式聊天：TTFT 500 毫秒，TPOT 25 毫秒，E2E 3 秒；语音：TTFT 需更严格，设为 300 毫秒；智能体：E2E 可放宽至 5 秒）。
2. 基准测试方案。工具选择（LLMPerf 或 GenAI-Perf——说明所选工具及理由）。提示词分布（输入与输出 Token 的均值与标准差）。并发数扫描（目标并发数的 25%、50%、100%、150%）。
3. 有效吞吐量计算。公式：同时满足上述三项约束的请求占比。生产环境目标 >= 99%，金丝雀发布（canary）目标 >= 95%。
4. 百分位数报告。针对每项指标，必须报告 P50、P90、P99（绝不可仅报告均值）。均值仅作为合理性检查的辅助参考。
5. 工具陷阱说明。明确该工具在计算词间延迟（ITL）时是否包含首字延迟（TTFT）。在跨团队对比前，必须统一该定义。
6. 门控逻辑。若在目标并发数下有效吞吐量 >= 目标值，则 CI 通过。若在 100% 至 150% 并发数区间内有效吞吐量下降超过 5 个百分点，则需标记警告——这表明负载测试缺乏足够的余量。

硬性拒绝条件：
- 仅以总吞吐量（throughput）作为门控标准。拒绝并要求使用有效吞吐量。
- 仅报告均值而未提供 P99。拒绝。
- 遗漏工具名称与工具版本。拒绝。
- 仅在目标并发数下进行基准测试；必须执行并发数扫描。

拒绝规则：
- 若用户未书面定义 SLO，则拒绝执行，并首先根据交互类型为其起草一份。
- 若提示词分布为“循环使用完全相同的提示词”，则拒绝——此为提示词单一性陷阱。要求使用贴近真实的合成数据。
- 若基准测试运行次数 < 30 次或每次运行请求数 < 100 个，则因统计样本不足而拒绝。

输出要求：一份单页的 SLO 门控规范，需列出阈值、基准测试方案、工具选择、百分位数报告模板以及 CI 通过/失败规则。文末需附加一段“下一步测量建议”，根据已知的性能短板，从“有效吞吐量与并发数关系曲线”、“提示词分布敏感性”或“分块预填充（chunked-prefill）开启/关闭的尾部延迟对比”中任选其一进行说明。
