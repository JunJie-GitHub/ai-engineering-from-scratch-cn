结合既定区域范围、数据驻留边界、预期的前缀缓存（prefix-cache）多样性以及首字延迟（TTFT）服务等级协议（SLA），制定一份多区域路由与灾难恢复（DR）方案。

输出内容：

1. **路由器选型**。选择具备缓存感知能力的路由器（如 vLLM Router 或 llm-d router），并描述 KV 事件通道。明确前缀哈希算法（例如 512-token 滚动哈希）及平局决胜规则（如队列深度最小优先）。
2. **路由策略**。采用区域优先策略，还是全局（GORGO 风格）最小化预填充（prefill）与往返时间（RTT）的策略？需结合提示词长度分布进行论证——长提示词（>8K tokens）可从跨区域路由中获益，而短提示词则不然。
3. **数据驻留分区**。在进行任何优化之前：明确出于法律合规要求（如 GDPR、HIPAA），哪些请求必须绑定至特定区域。即使跨区域路由能改善 TTFT，也严禁跨越驻留边界进行路由。
4. **商用 CRI 层**。建议是否启用 Bedrock Cross-Region Inference 或 GKE Multi-Cluster Gateway 作为可用性保障层。需明确指出该层并非用于优化 TTFT。
5. **灾难恢复清单**。至少包含三个文件（HF repo + 引擎配置 + 部署清单）。需验证是否包含分词器（tokenizer）、量化配置、旋转位置编码（RoPE）、对话模板及 LoRA 适配器。明确存储方案（如 S3 跨区域复制或多区域 GCS）。
6. **故障转移演练**。按季度执行。明确演练负责人及考核指标（恢复时间目标（RTO）、恢复点目标（RPO）、缓存预热时间）。目标：实现 30 分钟 RTO，以对标 2024 年摩根大通的实际演练标准。

**硬性拒绝条件**：
- 为优化路由而忽略数据驻留要求。必须拒绝——违反 GDPR 的代价远高于 TTFT 的收益。
- 声称 Bedrock CRI 能“解决”跨区域路由问题。必须拒绝——CRI 仅保障可用性，而非优化 TTFT。
- 仅备份模型权重。必须拒绝——需引用 32% 的灾难恢复失败率统计数据，并强制要求提供三文件清单。

**拒绝规则**：
- 若范围仅包含单一区域，则拒绝该方案——单区域架构的故障模式不同（详见第 17 阶段 · 第 03 课）。
- 若数据驻留要求与 TTFT SLA 存在冲突（例如：欧盟驻留要求导致每次请求均需在冷前缀上进行预填充，却要求 8K 提示词的 P99 TTFT < 100 ms），则拒绝承诺该 SLA，并将产品需求升级处理。

**输出要求**：一份单页方案，需明确路由器选型、路由策略、数据驻留分区、CRI 层部署姿态、灾难恢复清单及季度演练负责人。结尾需指定唯一告警指标：跨区域前缀缓存命中率低于方案设定的阈值。
