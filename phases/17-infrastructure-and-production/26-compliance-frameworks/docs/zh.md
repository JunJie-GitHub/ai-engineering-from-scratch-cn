# 合规性 — SOC 2、HIPAA、GDPR、PCI-DSS、欧盟《人工智能法案》（EU AI Act）、ISO 42001

> 多框架覆盖已成为2026年企业级交易的准入门槛。**欧盟《人工智能法案》（EU AI Act）**：自2024年8月1日起生效。大多数高风险相关要求将于2026年8月2日强制执行。违反高风险系统义务（第99条第4款）的罚款最高可达1500万欧元或全球年营业额的3%；违反禁止性AI实践（第99条第3款）的罚款最高可达3500万欧元或7%。若服务欧盟用户，该法案在全球范围内适用。**科罗拉多州人工智能法案（Colorado AI Act）**：2026年6月30日生效（因SB25B-004法案从2026年2月推迟）——要求对高风险系统进行影响评估，并赋予对AI决策提出申诉的权利。弗吉尼亚州在信贷、就业、住房和教育领域有类似规定。**SOC 2 第二类（SOC 2 Type II）**：B2B AI服务的实际强制要求（金融科技领域要求Type II而非Type I）。**《通用数据保护条例》（GDPR）**：有记录的最大AI专项罚款是针对Clearview AI的3050万欧元（荷兰数据保护局，2024年9月）；意大利数据保护局（Garante）于2024年12月对OpenAI处以1500万欧元罚款（后于2026年3月上诉中被推翻）。推理阶段的实时个人身份信息（PII）脱敏是合规抗辩的标准做法；仅靠后处理清理是不够的。**《健康保险流通与责任法案》（HIPAA）**：受医疗行业约束——若无商业伙伴协议（BAA），不得将受保护的健康信息（PHI）发送至外部AI服务。**支付卡行业数据安全标准（PCI-DSS）**：AI交互层的覆盖需要配置与合同协议，并非自动生效。**ISO 42001**：新兴的AI治理标准，正与ISO 27001一同成为日益增长的采购要求。参考案例：OpenAI 为 ChatGPT 支付组件维护 SOC 2 Type 2、ISO/IEC 27001:2022、ISO/IEC 27701:2019、GDPR/CCPA/HIPAA (BAA)/FERPA 以及 PCI-DSS 合规。跨框架映射可减轻审计疲劳：访问控制可映射至 ISO 27001 A.5.15-5.18、GDPR 第32条以及 HIPAA §164.312(a)。

## 学习目标

- 列举与大型语言模型（LLM）产品相关的七大2026年合规框架，并将其与相应的客户群体进行匹配。
- 引用欧盟《人工智能法案》（EU AI Act）的执行时间线（2024年8月生效；高风险要求于2026年8月强制执行）及两级罚款上限（高风险义务1500万欧元/3%，禁止性实践3500万欧元/7%）。
- 解释为何后处理个人身份信息（PII）清理不足以满足GDPR要求，并指出实时推理层脱敏是合规抗辩的标准做法。
- 描述跨框架控制映射（例如：访问控制映射至 ISO 27001 A.5.15-5.18 + GDPR 第32条 + HIPAA §164.312(a)）。

## 问题背景

某企业客户的采购部门要求提供 SOC 2 Type II、GDPR、HIPAA BAA、ISO 27001 以及“欧盟《人工智能法案》合规声明”。而你的团队目前仅持有 SOC 2 Type I 认证。距离获得 Type II 认证还有六个月，且尚未开始准备 GDPR 第30条要求的处理活动记录。

多框架覆盖并非大型语言模型（LLM）特有的问题，而是企业级SaaS的通用问题，仅叠加了LLM特定的要求。2026年的采购团队需要的是一份矩阵表格（每行对应一个框架，每列对应一项控制措施），而不是一份PDF文档。

## 核心概念

### 七大框架

| 框架 | 适用范围 | 大语言模型（LLM）特定要求 |
|-----------|-------|--------------------------|
| SOC 2 Type II（服务组织控制报告第二类） | B2B SaaS 基线 | 流程控制需经过 6-12 个月的审计 |
| HIPAA（健康保险流通与责任法案） | 美国医疗保健 | 必须签署商业伙伴协议（BAA）；受保护健康信息（PHI）未经签署协议不得离开基础设施 |
| GDPR（通用数据保护条例） | 欧盟用户 | 实时个人身份信息（PII）脱敏；数据主体权利；第 30 条记录 |
| PCI-DSS（支付卡行业数据安全标准） | 支付数据 | 涉及支付的 AI 需进行配置与合同管理 |
| 《欧盟人工智能法案》（EU AI Act） | 服务欧盟用户 | 风险分级；高风险系统：合格评定、文档记录、日志记录 |
| 《科罗拉多州人工智能法案》（Colorado AI Act） | 服务科罗拉多州居民 | 影响评估；申诉权 |
| ISO 42001（人工智能管理体系） | AI 治理 | 新兴标准；通常与 ISO 27001 配合使用 |

### 《欧盟人工智能法案》（EU AI Act）时间线

- 2024年8月1日：正式生效。
- 2025年2月2日：禁止类 AI 实践条款开始执行。
- 2026年8月2日：高风险系统条款开始执行（合格评定、文档记录、日志记录）。
- 2027年8月：受统一立法管辖产品中的高风险系统条款生效。

风险分级：不可接受（禁止）、高风险（合格评定 + 日志记录）、有限风险（透明度要求）、极低风险（无约束）。大多数 B2B LLM SaaS 属于有限风险；当应用于招聘、信贷、教育、执法、移民及基本公共服务领域时，将触发高风险要求。

罚款（第 99 条）：违反高风险系统义务（第 99(4) 条），最高罚款 1500 万欧元或全球年营业额的 3%；涉及禁止类 AI 实践（第 99(3) 条），最高罚款 3500 万欧元或全球年营业额的 7%；两者取其高者执行。

### GDPR —— 实时脱敏已成为标准

后处理清理（在 LLM 处理后再脱敏 PII）并非合规的防御姿态——因为模型已经接触到了数据。2026 年的标准是在推理层（inference layer）进行实时脱敏：

- 在调用 LLM 前进行实体识别。
- 采用一致的词元化（tokenization）方法（如 Mesh 方案）以保留语义。
- 仅存储已脱敏的提示词（prompt）及经用户明确同意保留的原始数据。

近期执法案例：针对 Clearview AI 的 3050 万欧元罚款（荷兰数据保护局，2024年9月）是迄今为止有记录的最大 AI 专项 GDPR 罚款；针对 OpenAI 的 1500 万欧元罚款（意大利数据保护局 Garante，2024年12月）是最大的 LLM 专项罚款，尽管该裁决在 2026年3月的上诉中被推翻且仍在进一步审查中。在审计中，主张“后处理即可合规”的说法已被否决。

### HIPAA —— 商业伙伴协议（BAA）并非可选项

在未签署商业伙伴协议（Business Associate Agreement, BAA）的情况下，不得将受保护健康信息（Protected Health Information, PHI）发送至外部 AI 服务。三大超大规模云厂商的 LLM 平台（Bedrock、Azure OpenAI、Vertex）均提供 BAA。OpenAI 直连 API 和 Anthropic 直连 API 也提供 BAA。在发送 PHI 前务必确认协议状态。

### SOC 2 Type II

Type I：控制措施已设计并文档化。
Type II：控制措施在 6-12 个月内持续有效运行。

2026 年的 B2B 采购默认要求 Type II。Type I 仅是入门；Type II 才是准入门槛。

常见审计驱动因素：访问日志（谁查看了哪些数据）、变更管理（部署流程）、风险评估（每季度）、事件响应（是否经过演练？）。第 17 阶段 · 25 的审计日志可直接复用。

### 跨框架映射

单一访问控制策略即可满足多个框架的控制要求：

| 控制项 | 适用框架 |
|---------|-----------|
| 访问日志记录 | ISO 27001 A.5.15-5.18, GDPR 第 32 条, HIPAA §164.312(a) |
| 变更管理 | ISO 27001 A.8.32, PCI DSS 要求 6, HIPAA 违规通知范围 |
| 传输中加密 | ISO 27001 A.8.24, GDPR 第 32 条, HIPAA §164.312(e) |
| 密钥/凭证管理 | ISO 27001 A.8.19, PCI DSS 要求 8, SOC 2 CC6.1 |

合规工具（如 Drata、Vanta、Secureframe）可自动完成此类映射。在规模化部署时，其投入成本物有所值。

### ISO 42001 —— 新兴标准

于 2023 年底发布。正逐渐成为与 ISO 27001 并列的采购要求。该框架涵盖 AI 治理，包括风险管理、数据质量、透明度及人工监督。

### OpenAI 的合规参考基准

OpenAI 目前持有 SOC 2 Type 2、ISO/IEC 27001:2022、ISO/IEC 27701:2019 认证，并符合 GDPR/CCPA/HIPAA（含 BAA）/FERPA 要求，其 ChatGPT 支付组件亦通过 PCI-DSS 认证。这大致构成了 2026 年企业级市场的基础门槛（table stakes）。

- 《欧盟人工智能法案》高风险系统执行日期：2026年8月2日。

## 练习

1. 你的首位企业客户要求提供 SOC 2 Type II、HIPAA BAA 以及《欧盟人工智能法案》（EU AI Act）声明。为赢得该订单，最低可行的合规态势（compliance posture）是什么？
2. 将三个假设的大语言模型（LLM）产品按照《欧盟人工智能法案》的风险等级（risk tiers）进行分类。在高风险（high-risk）级别下，要求有何变化？
3. 你意外将受保护的健康信息（PHI）发送给了未签署业务伙伴协议（BAA）的供应商。请逐步说明事件响应（incident response）流程。
4. 论证对于中型市场（mid-market）的 AI 供应商而言，ISO 42001 在 2026 年是否“必不可少”。
5. 将你的 LLM 审计日志（audit log）字段（第 17 阶段 · 25）映射到至少三项框架控制措施（framework controls）。

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------------|------------------------|
| SOC 2 Type II | “经过审计的控制措施” | 持续运行 6-12 个月的控制措施，并经独立机构鉴证 |
| HIPAA BAA | “医疗合同” | 业务伙伴协议（Business Associate Agreement）；处理 PHI 时的强制要求 |
| GDPR | “欧盟隐私法” | 实时个人身份信息（PII）脱敏是 2026 年可辩护的合规标准 |
| 《欧盟人工智能法案》（EU AI Act） | “欧盟 AI 法规” | 高风险条款于 2026 年 8 月生效；罚款 1500 万欧元或全球营业额 3%（高风险义务）—— 3500 万欧元或 7%（禁止类行为） |
| 《科罗拉多州人工智能法案》（Colorado AI Act） | “美国州级 AI 法律” | 2026 年 6 月 30 日生效（因 SB25B-004 法案推迟）；需进行影响评估 |
| ISO 42001 | “AI 治理” | 针对 AI 风险与透明度的新兴框架 |
| ISO 27001 | “安全 ISMS” | 信息安全管理体系（Information Security Management System）基线 |
| 符合性评估（Conformity assessment） | “欧盟 AI 文档包” | 高风险要求：文档、测试与日志记录 |
| 跨框架映射（Cross-framework mapping） | “一项控制，多个框架” | 单一策略即可满足多个框架的控制要求 |
