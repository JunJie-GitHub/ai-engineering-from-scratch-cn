# 批量 API（Batch APIs）—— 50% 折扣已成为行业标准

> 所有主流提供商均提供异步批量 API（Async Batch API），附带 50% 的折扣和约 24 小时的处理周期。OpenAI、Anthropic、Google 以及大多数推理平台（如 Fireworks 的批量层级、Together 的批量服务）都采用了相同的模式。将批量处理与提示词缓存（Prompt Caching）结合使用，可使夜间流水线的成本降至同步无缓存（Synchronous-Uncached）成本的约 10%。这条规则极其简单直接：只要不是交互式任务，就应该走批量处理。内容生成流水线、文档分类、数据提取、报告生成、批量标注、目录打标——任何能容忍 24 小时延迟的任务，在迁移到批量处理之前，都是在白白浪费预算。2026 年的生产模式是将每个新的大语言模型工作负载（LLM Workload）分流到三条通道中：交互式（带缓存的同步调用）、半交互式（带降级机制的异步队列）、批量处理（夜间运行，叠加缓存输入）。那些伪装成交互式但实际上能容忍分钟级延迟的工作负载，造成的浪费最为严重。

## 学习目标

- 列举三大提供商的批量 API（OpenAI、Anthropic、Google）及其通用的 50% 折扣与 24 小时处理周期保障。
- 计算在夜间分类工作负载上叠加批量处理与缓存输入的成本，并与同步无缓存基线进行对比。
- 将工作负载分流至交互式/半交互式/批量通道，并论证选择该通道的理由。
- 指出两大陷阱：部分交互性（用户期望的响应时间快于 24 小时）以及输出模式漂移（Output-Schema Drift，各提供商的批量文件格式存在差异）。

## 问题背景

你的团队正在交付一个夜间报告生成流水线。需要处理 50,000 份文档，逐一生成摘要，对摘要进行聚类，并起草一份高管简报。采用同步方式运行需要 4 小时，每晚成本为 2,000 美元。此时，你听说了批量 API。

批量处理能为你带来 50% 的折扣。你同时为系统提示词启用了提示词缓存（该提示词在所有 5 万次调用中共享）。两者叠加后，账单降至每晚 180 美元——仅为基线成本的约 9%。相同的流水线，仅修改了三项配置。

批量处理是大语言模型成本优化工具箱中最廉价却最常被忽视的杠杆。其原因主要在于组织架构：当服务等级协议（SLA）的实际要求只是“次日早晨交付”时，团队却习惯性地追求“实时响应”。本课程的核心就是教你如何避免白白浪费这 90% 的预算。

## 核心概念

### 三大批量 API (Batch APIs)

**OpenAI Batch API**：上传包含请求列表的 JSONL 文件。承诺 24 小时内完成（实际通常约 2-8 小时）。输入和输出 Token 享受 50% 折扣。使用 `/v1/batches` 端点。符合缓存条件的输入还可额外享受缓存输入定价。

**Anthropic Message Batches**：上传 JSONL 文件。24 小时处理周期。50% 折扣。支持 `cache_control` —— 缓存写入需显式指定，读取在批量处理中自动进行。

**Google Vertex AI Batch Prediction**：支持 BigQuery 或 GCS 输入。Gemini 模型同样享受约 50% 的折扣。可与 Vertex Pipelines 集成。

### 语义：异步而非缓慢

批量处理的承诺是“我保证在 24 小时内返回结果”，而不是“这需要 24 小时才能完成”。典型的第50百分位延迟 (P50) 为 2-6 小时。服务提供商会在 GPU 算力闲置的非高峰时段调度你的批量任务。

### 与缓存叠加使用

以使用相同 4K Token 系统提示词 (System Prompt) 的 5 万份文档摘要任务为例：

- 同步无缓存：50000 × ($input × 4000 + $output × 200)，按全价计费。
- 同步缓存：首次写入后系统提示词进入缓存；剩余 49999 次请求的输入成本降低至 1/10。
- 批量缓存：在上述基础上，读取和写入均额外享受 50% 折扣。

叠加效果：批量处理 + 缓存 ≈ 同步无缓存账单的 10%。任何可隔夜运行且共享系统提示词的工作负载都应采用此方案。

### 工作负载分级

**交互式 (Interactive)** —— 用户等待响应。首字生成时间 (TTFT) 至关重要。采用带提示词缓存的同步调用。无法使用批量处理。

**半交互式 (Semi-interactive)** —— 用户提交任务，几分钟后查看结果。使用异步队列，若批量处理不可用则降级为同步。适用于中等规模的检索增强生成 (RAG) 索引构建。

**批量式 (Batch)** —— 用户期望“明早”或“下一小时”拿到结果。内容处理流水线、大规模分类、离线分析。始终使用批量处理，并始终叠加缓存。

常见误区：因为流水线处于生产环境，就将所有任务都归类为交互式。生产环境本身不是延迟指标——服务等级协议 (SLA) 才是。

### 部分交互性陷阱

某些功能看似需要交互，但实际上可以容忍 5-10 分钟的延迟。例如：带有“刷新”按钮的每日夜间客户健康报告。用户点击刷新后，等待 10 分钟完全可以接受。但团队却将其作为同步接口上线。50 个并发刷新请求的成本，将是批量处理并通过邮件发送的 10 倍。

需要问的问题是：“24 小时的处理周期对该用户意味着什么？”如果答案是“他们根本察觉不到”，那就使用批量处理。

### 输出结构陷阱

各提供商的批量文件格式各不相同：

- OpenAI：JSONL 格式，每行一个请求。
- Anthropic：JSONL 格式，每行一条消息；响应格式已嵌入。
- Vertex：BigQuery 表或带有 TFRecord 的 GCS 前缀。

编写跨提供商的“统一批量客户端”意味着需要为每个提供商编写适配代码。那些宣称支持多提供商批量处理的网关（如 Portkey、LiteLLM 的某些层级）仍然只是对原始格式进行了轻量级封装。

### 需要牢记的关键数据

- 各提供商批量折扣：输入 + 输出统一享受 50% 折扣。
- 处理周期 SLA：保证 24 小时内完成，典型 P50 为 2-6 小时。
- 批量处理 + 缓存输入叠加：成本约为同步无缓存的 10%。
- 工作负载分级原则：若可接受 24 小时延迟，始终优先使用批量处理。

## 核心术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------------|------------------------|
| 批处理 API（Batch API） | “异步折扣” | 承诺 24 小时内返回，享受 50% 折扣 |
| JSONL | “批处理格式” | 每行一个 JSON 请求；OpenAI/Anthropic 标准格式 |
| Message Batches | “Anthropic 批处理” | Anthropic 的批处理 API 产品名称 |
| 批量预测（Batch prediction） | “Vertex 批处理” | Vertex AI 的批处理 API 产品 |
| 周转 SLA（Turnaround SLA） | “24 小时承诺” | 属于服务保证而非典型耗时；典型处理时间为 2-6 小时 |
| 工作负载分类（Workload triage） | “交互性决策” | 决定将请求路由至交互式、半交互式或批处理模式的决策 |
| 输出模式（Output schema） | “响应格式” | 各云厂商特定的 JSONL 布局；不具备跨平台可移植性 |
| 叠加折扣（Stacked discount） | “批处理 + 缓存” | 当两者同时生效时，成本约为未使用缓存的同步调用账单的 10% |
