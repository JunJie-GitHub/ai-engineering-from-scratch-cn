给定工作负载（端点、首字延迟 (TTFT)/每令牌输出时间 (TPOT)/错误率的服务等级协议 (SLA)）、目标规模（并发数、每秒请求数 (RPS)）以及持续集成 (CI) 策略（拉取请求 (PR) 门禁或仅发布阶段运行），请制定一份负载测试计划。

产出内容：

1. 工具。使用 LLMPerf 进行基线测试；使用 k6 配合流式扩展插件进行 CI 门禁测试；使用 GenAI-Perf 进行 NVIDIA 参考基准测试；使用 guidellm 进行大规模合成数据测试。仅当团队已在使用 Locust 时才考虑 LLM-Locust。
2. 提示词分布。基于真实流量（如有）或公开数据集（ShareGPT / HumanEval）的输入令牌均值与标准差。禁止使用单一提示词循环测试。
3. 四种测试模式。稳态 (Steady)、爬坡 (Ramp)、尖峰 (Spike)、浸泡 (Soak)。针对每种模式需明确：目标 RPS、持续时间、预期故障模式。
4. CI 门禁。具体阈值：TTFT P95 < X，5xx 错误率 < 5%，TPOT < Y。每次 PR 的运行时间：3-5 分钟。
5. 指标对齐。需注明报告工具采用的是 GenAI-Perf 风格（令牌间延迟 (ITL) 不包含 TTFT）还是 LLMPerf 风格（ITL 包含 TTFT）。选定一种并保持全程一致。
6. 输出物。提交至代码仓库的脚本文件（k6 JS 脚本或 LLMPerf CLI 命令）。

硬性拒绝条件：
- 使用统一提示词进行负载测试。拒绝——数据会失真。
- 不支持流式传输的负载测试。拒绝——LLM 端点默认采用流式输出。
- 在未说明指标定义差异的情况下跨工具对比数据。拒绝。

拒绝规则：
- 若团队计划使用原生 Locust 且未安装 LLM-Locust 扩展，拒绝——会触发全局解释器锁 (GIL) 陷阱。
- 若每次 PR 的 CI 门禁时间预算不足 60 秒，拒绝执行完整的浸泡测试——建议改为快速稳态测试，并另行安排夜间浸泡测试。
- 若无法获取提示词分布数据，则要求使用有文档记录的公开分布数据（如 ShareGPT），并明确标注该假设。

输出要求：一份单页计划，需包含工具选择、提示词分布、带目标的四种测试模式、CI 门禁阈值及指标对齐说明。最后明确唯一的 CI 输出标准：仅当所有阈值均达标且连续 3 次运行保持稳定时，PR 状态才显示为通过（绿色）。
