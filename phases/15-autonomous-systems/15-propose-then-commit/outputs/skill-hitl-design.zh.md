针对给定的 HITL 工作流提案，对照“先提议后提交”参考规范进行审计，并标记缺失、定义不足或与监管要求不兼容的部分。

输出内容：

1. **提案元数据**。确认每个提案均明确包含：意图（intent）、数据血缘（data lineage）、涉及的权限、爆炸半径（blast radius）以及回滚计划（rollback plan）。缺失字段即为阻碍项；“智能体想要执行 X"不构成有效提案。
2. **幂等性**。明确幂等键（idempotency key）的构成方式。该键必须能从提案内容中推导得出，以确保重试时返回相同的记录。包含物理时钟时间（wall-clock time）的键不属于幂等键，它们仅是日志时间戳。
3. **持久性**。明确存储介质（如 PostgreSQL、Redis、Durable Object 或带完整性校验的对象存储）。确认审批记录在智能体重启、主机重启及部署后依然存活。内存队列不符合要求。
4. **审批界面**。形式化审批（rubber-stamp approval，即单个批准按钮）无法通过此审计。必需项：包含质询-响应清单，要求对意图理解、爆炸半径验证及回滚准备情况进行明确确认。确认该清单是针对特定操作类别定制的，而非通用模板。
5. **提交后验证**。确认工作流在执行后会重新读取目标资源，并在验证失败时发出告警。“工具返回 200 状态码”不等于验证通过。

硬性拒绝条件：
- 未能持久化保存提案的 HITL 界面。
- 审核者即为智能体自身的审批流程。
- 任何缺乏质询-响应机制的不可逆生产环境操作。
- 包含物理时钟时间组件的幂等键。
- 在关键操作后缺失提交后验证的工作流。

拒绝规则：
- 若用户仅能命名审批界面，却无法指明其背后的持久化存储，则予以拒绝，并要求先明确存储方案。
- 若用户将 `max_budget_usd` 加确认对话框视为充分的 HITL 机制，则予以拒绝。预算仅限制成本，不保证正确性。
- 若部署涉及欧盟高风险范围且仍采用形式化审批模式，则依据《欧盟人工智能法案》第 14 条予以拒绝。

输出格式：

返回一份“先提议后提交”审计报告，包含以下内容：
- **提案字段表**（意图 / 血缘 / 爆炸半径 / 回滚 / 权限 — 五项均为必填）
- **幂等性说明**（键构成方式、重试测试结果）
- **持久性说明**（存储介质、重启后是否存活 y/n）
- **审批界面**（形式化审批 / 清单；若为清单，请列出具体问题）
- **提交后验证**（是否存在 y/n、重新读取的内容）
- **就绪状态**（生产环境 / 预发环境 / 仅限研究）
