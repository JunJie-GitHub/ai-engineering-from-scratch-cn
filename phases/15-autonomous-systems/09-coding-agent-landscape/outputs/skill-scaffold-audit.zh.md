针对拟采用的编码智能体脚手架（coding-agent scaffold）（如 SWE-agent、OpenHands、Aider、Cline、Devin、Claude Code 或内部自研版本），请从四个维度对其进行评分，并标出基准测试（benchmark）数据可能高估生产环境质量的环节。

请生成：

1. **检索（Retrieval）。** 描述该脚手架如何筛选智能体在采取行动前需要读取的文件。是通过仓库地图（repo map）、嵌入向量搜索（embedding search）、显式文件列表，还是由智能体驱动的 `grep` 调用？检索质量是决定系统可靠性的隐性核心因素。
2. **验证器循环（Verifier loop）。** 该脚手架是否会运行测试、读取堆栈跟踪（stack trace），并将失败信息反馈至下一轮交互？若缺少验证器循环，请标记为缺失——在类似 SWE-bench 的任务中，这通常会导致绝对分数出现 10 分以上的差距。
3. **沙箱与影响范围（Sandbox and blast radius）。** 操作在何处执行？本地文件系统、临时容器（ephemeral container）还是托管虚拟机（managed VM）。对于 CodeAct 风格的脚手架，需确认沙箱已进行安全加固（禁止出站流量、无主机挂载、设置超时限制）。对于 JSON 工具调用（JSON tool-call）脚手架，需确认工具验证器能够拦截所有非预期的副作用。
4. **基准测试适配度（Benchmark fit）。** 所报告的分数（例如“在 SWE-bench Verified 上达到 80.9%”）实际覆盖了怎样的数据分布？统计基准测试中由 1-2 行代码任务所占的比例；将报告的分数与同一模型在 SWE-bench Pro（10 行以上代码任务）上的得分进行对比。若一个脚手架的核心指标主要由简单尾部数据（easy tail）拉高，则该数据不能作为生产环境的参考依据。

硬性否决条件：
- 用于处理超出基础复杂度任务的任何脚手架，若缺少验证器循环，一律否决。
- 指向真实代码仓库的 CodeAct 脚手架，若缺乏沙箱隔离机制（无 Docker、无无根容器、无虚拟机），一律否决。
- 未公开数据分布（简单尾部任务占比、Pro 等效分数）的基准测试声明，一律否决。
- 工具调用脚手架中，若存在单个工具可在无验证器的情况下访问任意路径（例如向模型暴露原始的 `shell_exec` 工具），一律否决。

拒绝规则：
- 若用户无法提供该脚手架在具有代表性的内部数据分布上的测试套件通过率，则予以拒绝，并要求先进行小样本测量。公开基准测试仅能预测相对排名，无法反映绝对质量。
- 若拟采用的脚手架将在未经过预发环境（staging）试运行（dry-run）的情况下直接对接生产环境代码仓库，则予以拒绝，并要求先在预发环境进行测试。编码智能体会重写文件；检索能力差的编码智能体会重写错误的文件。
- 若用户计划仅依赖基准测试分数（而不使用内部评估数据）来做出推进/终止决策，则予以拒绝，并要求提供内部评估（internal eval）数据。

输出格式：

返回一份带评分的备忘录，包含以下内容：
- **检索得分**（0–5 分，需描述具体机制）
- **验证器循环得分**（0–5 分，需说明反馈格式）
- **沙箱得分**（0–5 分，需说明隔离机制）
- **基准测试适配度得分**（0–5 分，需说明内部数据分布差异）
- **部署建议**（生产环境 / 预发环境 / 仅限研究）
- **单行风险摘要**（最可能首次在生产环境中出现的故障）
