针对某一任务类别（task class，即重复执行的智能体运行类型，例如“重构函数”或“关闭支持工单”），维护一个反思的情景记忆缓冲区（episodic-memory buffer）。每条反思均以自然语言记录失败模式（failure mode）及相应的纠正性洞见。该缓冲区的内容将作为前置提示，附加到同一任务类别的下一次试验（trial）中。

产出要求：

1. 反思捕获（Reflection capture）。当试验结束且评估器（evaluator）得分低于阈值时，生成一条单行反思，格式为“我未能完成 X，原因是 Y；下次我将 Z。”除非外部故障（如网络问题、上游 500 错误）可复现，否则应丢弃相关反思。
2. 生存时间（TTL）与去重（dedup）。默认情况下，反思在经过 N 次试验后过期（建议设为 10）。完全相同的重复项将被合并。近似重复项（在小型嵌入模型（embedding model）上余弦相似度 >0.9，或共享子串 >= 80%）仅保留最新的一条。
3. 作用域策略（Scope policy）。包含三种作用域：任务类别（按任务名称划分）、用户（同一用户跨任务）、智能体（跨所有用户）。默认采用任务类别作用域。仅当反思涉及用户特定偏好时，才升级至用户作用域；切勿自动升级至智能体作用域。
4. 压缩（Compaction）。当缓冲区超出容量预算时，执行休眠期压缩（sleep-time compaction）：对近似重复项进行聚类、摘要与合并。压缩操作应在热路径（hot path）之外运行，切勿延迟主智能体的响应。
5. 提示词集成（Prompt integration）。输出一个标题为“我从过往试验中学到了什么”的独立区块，内含项目符号列表。提示词中最多保留 6 条；超出部分归入单独的摘要项中（例如“……以及 4 条关于超时的较早反思”）。

硬性拒绝项：

- 将反思写成“下次要更小心”之类的空话。此类内容不具备可操作性。需使用强制要求提供具体“下次操作指令”的提示词重新运行反思器（reflector）。
- 基于物理时钟时间（wall-clock time）而非试验次数来使反思过期。为支持离线可重放运行，TTL 应以试验次数为作用域，而非时间作用域。
- 存储引用了敏感信息（如 API 密钥、令牌、个人身份信息 PII）的反思。在提交至缓冲区前，必须拦截并返回明确的“包含敏感信息”类错误。

拒绝执行规则：

- 若未附加评估器，则拒绝执行并推荐学习第 05 课（Self-Refine/CRITIC）——反思需要明确的反馈信号，而非主观直觉。
- 若任务类别为一次性任务（one-shot，永不重复），则拒绝执行；情景记忆对从不重复的任务毫无作用。

输出：一个结构化缓冲区文件（JSON 格式，包含反思对象：试验 ID、任务类别、作用域、文本、创建时间、剩余 TTL）、用于下一次试验的提示词区块，以及一份“过期反思”报告（列出即将过期的条目）。
