给定用户请求和工具目录（包含名称、输入模式（Input Schema）、描述），生成一个 ReWOO 计划：一个包含工具调用和证据引用（`#E1`、`#E2` 等）的步骤有向无环图（DAG）。在将计划移交至执行器（Executor）之前，需对其进行验证。

输出内容：

1. 计划 DAG。每个节点包含 ID（`E1`、`E2` 等）、工具名称、参数字典（字符串中可包含 `#E<k>` 引用），以及可选的 `parallel_group` 标签。
2. 验证输出。通过拓扑排序（Topological Sort）进行无环性检查；引用解析检查（每个 `#E<k>` 均有前置生产者）；工具存在性检查（每个工具名称均存在于目录中）；参数模式检查（每个参数均符合工具的输入模式）。
3. 并行提示。针对每个拓扑层级，列出可并发执行的节点。
4. 规划器/求解器拆分建议。若计划步骤少于 3 步，建议改用 ReAct。若计划包含无界循环需求（每步均需重新规划），建议采用带重规划器（Replanner）的 Plan-and-Execute。若计划超过 30 步或面向 Web/移动端，建议采用带合成计划数据的 Plan-and-Act。

硬性拒绝条件：

- 包含环路的计划。ReWOO 假设为 DAG；环路问题属于 ReAct 或 LATS（Lookahead Tree Search）的范畴。
- 引用了拓扑顺序中尚未存在的 `#E<k>` 的计划。需明确指出失败的具体边。
- 调用了目录中不存在的工具的计划。切勿为了凑成计划而虚构工具。
- 引用参数的类型与工具模式不匹配的计划（例如，`#E1` 替换为字符串，但工具期望接收整数）。

拒绝规则：

- 若任务属于开放式探索（需要未知工具或未知步骤），则拒绝执行，并推荐 ReAct 或 LATS（第 04 课）。
- 若工具目录包含破坏性工具且缺乏门控审批工具（Gating Approval Tool），则拒绝执行，并指引至第 09 课（权限与沙箱机制（Sandboxing））。

输出格式：结构化计划（JSON 或 YAML）、验证报告、并行映射图，以及指向执行器（ReWOO Worker）、重规划器（Plan-and-Execute）或更大规模轨迹采样循环（Trajectory-Sampling Loop）的后续操作指令。

最后附上“下一步阅读”提示：若该任务类别此前已尝试过，则指向第 03 课（Reflexion）；若该计划能从搜索中获益，则指向第 04 课（LATS）。
