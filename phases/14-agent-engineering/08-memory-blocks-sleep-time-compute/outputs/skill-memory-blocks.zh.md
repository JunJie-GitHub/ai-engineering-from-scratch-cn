给定目标运行时（runtime）、主模型以及一个（可能更强的）休眠期模型，构建一个具有明确块类型和异步整合（consolidation）功能的三层记忆系统。

产出内容：

1. `Block` 类型，包含 `label`、`value`、`limit`、`description`、`version`、`history` 字段。每次写入都会递增版本号并记录旧值。暴露 `near_limit(threshold=0.8)` 方法。
2. 一个 `BlockStore`，至少包含三个默认块：`human`（用户相关事实）、`persona`（智能体自我认知）和 `task`（当前任务范围）。允许用户自定义块。
3. 一个 `Recall` 存储——按会话分页的交互轮次日志。每轮自动写入。达到容量上限时尾部数据会被逐出（evict），但仍保持可检索状态。
4. 一个 `Archival` 存储——至少支持两种后端（向量、键值对）。插入操作返回记录 ID。遇到冲突时执行失效（invalidate）而非删除。
5. 一个 `PrimaryAgent`，负责处理交互轮次并仅执行原始写入。关键路径上禁止进行摘要（summarization）操作。
6. 一个 `SleepTimeAgent`，在轮次间隙运行：对超过阈值的块进行摘要，使冲突的归档记录失效，并将 `learned_context` 写入共享块。

硬性拒绝条件：

- 除直接查找外，任何在面向用户的交互轮次中同步运行的记忆操作。摘要、整合、失效操作均应归属到休眠期处理阶段。
- 在发生冲突时删除归档记录。应执行失效操作以确保历史记录可审计。
- 未经审查步骤直接向 `Persona` 或 `Safety` 块写入数据。这些块全局性地塑造智能体行为；静默写入会掩盖缺陷。

拒绝规则：

- 如果运行时无法在会话间持久化块数据，则拒绝交付标称为“记忆”的产品。需降级相关声明。
- 如果休眠期智能体没有追踪（trace）输出，则拒绝。静默整合会导致调试盲区。
- 如果用户要求“不执行失效，始终信任最新写入”，则在历史记录至关重要的领域（合规、医疗、法律）必须拒绝。

输出要求：每个组件对应一个独立文件，外加一个 `README.md`，其中需明确默认块名称、休眠期执行频率以及冲突解决策略。结尾需包含“下一步阅读”指引：若智能体需要基于记忆进行图推理（graph reasoning），则指向第 09 课；若产品需要在记忆操作上集成 OTel 跨度（spans），则指向第 23 课。
