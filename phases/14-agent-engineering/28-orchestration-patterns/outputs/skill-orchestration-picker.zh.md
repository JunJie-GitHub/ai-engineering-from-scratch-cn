给定产品领域与任务类别，选择最简拓扑结构（Topology）。

决策：

1. 单个智能体（Agent）结合工作流模式（Workflow Patterns，第12课）是否已足够？ -> 完全无需使用拓扑结构。
2. 是否存在2-4个职责明确的专业智能体？ -> **监督者-工作者（Supervisor-Worker）**。
3. 对延迟敏感且专业智能体能够清晰交接？ -> **群体（Swarm）**。
4. 拥有10个以上专业智能体，且监督者的上下文预算（Context Budget）即将耗尽？ -> **分层（Hierarchical）**。
5. 准确性优先于成本，且多提案者+批判机制有效？ -> **辩论（Debate）**（第25课）。

产出：

1. 所选拓扑结构的脚手架（Scaffold）。
2. 群体结构需配置跳数计数器（Hop Counter）；分层结构需设置嵌套深度限制（Nesting Depth Limit）；辩论结构需设定轮次上限（Round Cap）。
3. 针对每次交接或每一步骤的可观测性钩子（Observability Hooks）（OTel GenAI 跨度/Span，第23课）。
4. README 文件中需包含“为何选择此方案而非彼方案”的说明章节。

硬性拒绝条件：

- 将连续3次大语言模型（LLM）调用称为“多智能体”。那仅是提示词链（Prompt Chain）。
- 未配置跳数计数器的群体结构。必然会导致请求循环反弹（Bouncing）。
- 每个分支最终仅收敛至1个专业智能体的分层结构。应将其扁平化（Flatten）处理。

拒绝规则：

- 若用户希望为单个 ReAct 循环（ReAct Loop）即可处理的任务引入多智能体，应予以拒绝并建议参考第01课。
- 若用户希望为仅需2步的任务使用监督者模式，应予以拒绝并建议采用提示词链（第12课）。
- 若该领域存在合规/审计要求，应拒绝群体结构，并建议采用监督者或分层结构。

输出：拓扑结构脚手架 + 包含决策依据的 README。结尾需附上“下一步阅读建议”，指向第13课（LangGraph）用于监督者实现，第16课（OpenAI Agents SDK）用于将交接作为工具（Handoffs-as-Tools），或第25课用于辩论机制的具体细节。
