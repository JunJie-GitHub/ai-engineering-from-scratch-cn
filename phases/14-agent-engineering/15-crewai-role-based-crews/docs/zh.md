# CrewAI：基于角色的智能体组（Crews）与工作流（Flows）

> CrewAI 是 2026 年推出的基于角色的多智能体框架（Multi-Agent Framework）—— 智能体（Agent）、任务（Task）、智能体组（Crew）与流程（Process）构成其四大原语（Primitives）。官方文档针对生产环境的建议是：“对于任何生产就绪型应用，请从工作流（Flow）开始。”

**类型：** 学习 + 实战
**语言：** Python（标准库）
**前置知识：** 第 14 阶段 · 12（工作流模式），第 14 阶段 · 14（Actor 模型）
**预计耗时：** 约 60 分钟

## 学习目标

- 说出 CrewAI 的四大原语——智能体、任务、智能体组与流程——并阐明各自的作用。
- 区分智能体组（基于角色的自主协作）与工作流（事件驱动的确定性工作流）。
- 解释为何官方文档建议在生产环境中优先使用工作流，而在探索性任务中使用智能体组。
- 使用标准库实现一个智能体组运行器（Crew runner）和一个工作流运行器（Flow runner）；展示各自的最佳适用场景。

## 问题背景

采用多智能体框架的团队往往会遇到同样的瓶颈：“自主协作”听起来很美好，但当客户提交 Bug 时，你需要的是确定性的可重现执行。CrewAI 对此进行了明确划分——智能体组用于创造性协作，工作流用于事件驱动、可审计且符合生产规范的工作流。

## 核心概念

### 四大原语

- **智能体（Agent）。** 角色 + 目标 + 背景故事 + 工具。背景故事是核心支撑要素——它决定了智能体的语气与判断逻辑。
- **任务（Task）。** 描述 + 预期输出（expected_output）+ 指定智能体。可复用的工作单元。
- **智能体组（Crew）。** 用于编排智能体与任务的容器。负责管理执行流程。
- **流程（Process）。** 支持顺序执行、层级执行（含管理智能体）或共识执行。

### 智能体组与工作流对比

- **智能体组（Crew）。** 自主运行，由大语言模型（LLM）驱动。适用于开放式任务：如资料调研、头脑风暴、初稿撰写等。框架会在运行时动态决定执行形态。
- **工作流（Flow）。** 事件驱动，由代码定义的图结构。每个步骤由触发器（如函数装饰器、事件匹配）激活。适用于生产环境：具备可观测性、可测试性与确定性。

CrewAI 2026 版文档指出：生产级应用应从工作流起步；仅当自主协作带来的收益足以覆盖其成本时，再将智能体组作为子步骤嵌入其中。

### 记忆系统

CrewAI 开箱即提供四种记忆类型：短期记忆（单次运行内）、长期记忆（跨运行）、实体记忆（针对特定实体的事实）与上下文记忆（检索时动态组装）。官方原生支持向量数据库（Vector Stores）集成。

### AWS Bedrock 集成

CrewAI 提供了与 AWS Bedrock 集成的官方文档，并内置了 CloudWatch、AgentOps 和 Langfuse 的可观测性钩子（Observability Hooks）。AWS 官方文档在其基准测试中指出，在问答（QA）任务上其速度比 LangGraph 快 5.76 倍——请注意，此类框架专属的性能数据仅作参考方向，而非绝对标准。

### 依赖与环境

独立于 LangChain。支持 Python 3.10–3.13。使用 `uv` 进行依赖管理。截至 2026 年初，GitHub 星标数已突破 3 万。

### 常见误区

- **将智能体组直接用于生产环境。** 在没有工作流封装的情况下，直接将自由形态的智能体组投入生产。输出变异性极高，调试极其困难。
- **背景故事冗长。** 长达 2000 字的背景故事会挤占上下文窗口预算。应保持精简。
- **流程模式误用。** 层级流程（Hierarchical Process）会引入一个负责路由的管理智能体；仅在拥有 4 个及以上专业智能体时才建议使用。

## 动手实战

`code/main.py` 实现了两者的标准库 (stdlib) 版本：

- `Agent`、`Task`、`Crew`、`SequentialCrew`（顺序执行，一次处理一个任务）、`HierarchicalCrew`（层级路由，由管理器分配）。
- `Flow`，带有 `@start()` 和 `@listen()` 装饰器 (decorators)（作为普通函数的占位符），在触发命名事件时执行。
- 相同的三步任务（调研、大纲、起草）分别用这两种方式实现。

运行方式：

python3 code/main.py

`Crew` 的执行轨迹 (trace) 动态且多变；`Flow` 的执行轨迹固定且可观测。这就是你需要做出的选择。

## 使用建议

- 生产环境使用 **CrewAI Flow**。
- 探索、结对编程或生成初稿使用 **CrewAI Crew**。
- 如果需要更明确的状态机 (state machine)，请使用 **LangGraph**（第 13 课）。
- 如果需要基于 Actor 模型 (actor-model) 的并发能力，请使用 **AutoGen v0.4**（第 14 课）。

## 练习

1. 将基于 Crew 的演示程序转换为 Flow。统计可变性降低的交互节点 (touchpoints) 数量。
2. 为 Crew 添加实体记忆 (entity memory)：使关于客户的事实信息能够在不同任务间持久化。
3. 实现层级流程 (Hierarchical process)：由管理器智能体 (manager Agent) 根据上一阶段的输出，决定下一步运行哪个专家智能体。
4. 阅读 CrewAI 官方文档的简介部分。将你的原型程序迁移至正式的 `crewai` API。可测试性 (testability) 发生了哪些变化？
5. 将 AgentOps 或 Langfuse 接入你的某次运行中。在标准库版本中，你遗漏了哪些执行轨迹 (traces)？

## 核心术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------------|------------------------|
| Agent | “角色设定” | 角色 + 目标 + 背景故事 + 工具 |
| Task | “工作单元” | 描述 + 预期输出 + 负责人 |
| Crew | “智能体团队” | 智能体 + 任务 + 流程的容器 |
| Process | “执行策略” | 顺序执行 / 层级路由 / 共识决策 |
| Flow | “确定性工作流” | 事件驱动、由代码控制、可测试 |
| Backstory | “角色提示词” | 塑造智能体语气与判断力的背景设定 |
| Entity memory | “单实体事实” | 作用域限定于特定客户/账户/问题的记忆 |
