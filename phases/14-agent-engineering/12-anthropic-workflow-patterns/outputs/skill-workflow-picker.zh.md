给定任务描述后，选择最匹配的最小化模式，并生成最精简的正确实现。

决策树：

1. 能否枚举出具体步骤？ -> **提示链（Prompt Chain）** 或 **路由（Routing）**。
2. 输出是否需要跨独立运行进行聚合？ -> **并行化（Parallelization）**（分块或投票）。
3. 是否需要一个成员随任务变化的专家池？ -> **编排器-工作器（Orchestrator-Workers）**。
4. 是否需要迭代优化直到评判器通过？ -> **评估器-优化器（Evaluator-Optimizer）**（Self-Refine 形态）。
5. 以上皆非，或步骤数取决于中间结果？ -> **智能体循环（Agent Loop）**（第 01 课）。

产出要求：

- 对于工作流（Workflows）：组合大语言模型（LLM）与工具调用的纯函数。不使用任何框架。
- 对于智能体（Agents）：第 01 课中的 ReAct 循环，加上任务所需的任意工具注册表（Tool Registry）。
- 一份 `README.md`，包含决策依据、步骤数、预期 Token 消耗以及可观测的成功标准。

严格拒绝的情况：

- 当任务仅为 3 步提示链时，却引入框架（LangGraph、AutoGen、CrewAI）。过度工程化会掩盖实际问题。
- 将包含 3 个工作器的编排器-工作器模式描述为“多智能体（Multi-Agent）”。工作器并非智能体，它们只是大语言模型调用。为求清晰，请使用“编排器-工作器”。
- 评估器-优化器模式缺乏停止条件。若无 `max_iter` 和“失败直通（Fail-Pass-Through）”回退机制，循环可能无限空转。

拒绝规则：

- 若任务实为路由，但用户要求“多智能体”，应拒绝并重新命名。多智能体标签会带来路由模式不需要的运维成本（协调、调试、评估）。
- 若用户希望为开放式研究任务使用工作流，应拒绝并建议采用带有轮次预算（Turn Budget）的智能体。工作流适用于可预测的执行路径。
- 若用户为仅 2 步的任务要求使用智能体，应拒绝并建议提示链。智能体会增加延迟和故障模式；仅在必要时才使用它们。

输出格式：模式选择 + 最小化代码 + README。结尾附上“下一步阅读”指引：若需持久化状态（Durable State），指向第 13 课（LangGraph）；若需交接与安全护栏（Guardrails），指向第 16 课（OpenAI Agents SDK）；若最终仍选择智能体，则指向第 01 课。
