给定产品领域与轨迹存储库（trace store），生成用于检测智能体（agent）故障模式（failure modes）的检测器。

产出内容：

1. 针对每种模式的检测器：`hallucinated_action`、`scope_creep`、`cascading_errors`、`context_loss`、`tool_misuse`、`success_hallucination`。
2. 特定领域检测器（例如：针对开发工具的“创建 PR 但未关联 Issue”，针对营销工具的“未经确认向超过 5 名收件人发送邮件”）。
3. 标记器（Tagger）：将全部检测器应用于每条轨迹，并输出分布结果。
4. 基于阈值的告警：若今日轨迹中 >=5% 标记了某种模式，则触发寻呼（page）或创建工单。
5. 样本留存：针对每条被标记的轨迹，保留输入、输出及状态快照，供运维人员审查。

严格拒绝项：

- 在生产环境中要求对每条轨迹调用大语言模型（LLM）的检测器。请使用基于模式的检测器；将 LLM 评判（LLM-judge）仅保留用于抽样审查。
- 仅在系统崩溃时进行标记。大多数故障会产生看似有效的输出。必须对内容与状态进行特征签名检查。
- 存储未经个人身份信息（PII）脱敏的标记轨迹。故障样本往往包含最敏感的内容；存储前必须进行数据清洗。

拒绝规则：

- 若用户要求“永久存储所有轨迹”，出于成本与合规原因予以拒绝。应按标签与采样率进行抽样。
- 若产品缺乏“已知良好”基线，拒绝漂移（drift）告警。漂移检测需要参考基准。
- 若检测器未进行版本控制，予以拒绝。检测器回退会在毫无预警的情况下破坏信号质量。

输出文件：`detectors.py`、`tagger.py`、`alerts.py`、`retention.py`、`README.md`（需说明阈值、留存策略及告警路由）。结尾需包含“下一步阅读”指引，指向第 24 课（可观测性后端）或第 27 课（提示词注入），以探讨对抗性故障模式。
