针对拟定的工具定义，在工具交付给模型之前，对照四步循环（描述、决策、执行、观察）进行审查，并标记可能破坏循环的缺陷。

生成内容：

1. 名称审查。名称是否为 `snake_case`，跨版本保持稳定且无歧义？标记与内置项冲突、包含时态（如 "was_"、"will_"）或嵌入参数的名称。
2. 描述审查。描述是否读起来像一份完整的使用简报？要求采用两句话的格式：“在 X 情况下使用。不要用于 Y。”标记少于 40 个字符的描述、营销话术或任何无法指导模型进行选择的内容。
3. 模式审查。该模式是否为有效的 JSON Schema 2020-12？每个字段是否都有类型定义？`required` 列表是否明确？封闭值集是否使用了枚举（Enums）？标记本应使用枚举的开放式字符串字段、缺失的类型定义，以及输入对象中未声明的 `additionalProperties`。
4. 执行器审查。给定参数时，执行器是否具有确定性（deterministic）？是否使用类型化错误（typed error）处理失败（而不是抛出逃逸宿主环境的异常）？如果该工具具有后果性（consequential）（改变状态、产生费用或涉及用户数据），是否已作相应标记并置于确认门控之后？
5. 分类。说明该工具是纯函数型（pure）还是后果型（consequential）及其原因。未设置确认门控的后果型工具将直接被拒绝。

硬性拒绝条件：
- 任何描述仅说明“做什么”而未说明“何时使用”的工具。模型在第二步需要“何时”的信息。
- 任何包含未定义类型字段的模式。验证器（validator）将无法正常工作。
- 同时满足以下三项的工具：接受不可信输入、读取敏感数据、执行后果性操作。这违反了 Meta 的“双重规则”（Rule of Two）。
- 任何在输入错误时执行器会抛出未处理异常的工具。宿主环境不应在每次调用时都包裹 `try/except`。

拒绝规则：
- 如果工具定义缺少模式，则拒绝。请先转至第 13 阶段 · 04 课。
- 如果工具是纯函数型，但描述中写着“谨慎使用”，则拒绝并要求说明原因。纯函数型工具的重运行成本应当很低。
- 如果要求审查员批准一个未设置只读保护却直接连接生产数据库的工具，则拒绝，并指引至第 13 阶段 · 17 课（网关与策略（gateways and policy））。

输出：一份单页审查报告，列出名称、描述、模式和执行器的审查结果，并标注严重程度（阻断 / 警告 / 建议），最终给出发布 / 修改 / 拒绝的结论。如果可行，在末尾为任何被拒绝的工具提供一行重写建议。
