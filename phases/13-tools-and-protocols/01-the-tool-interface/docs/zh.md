# 工具接口（Tool Interface）—— 为什么智能体（Agent）需要结构化 I/O

> 语言模型生成的是词元（token）。程序执行的是动作。两者之间的鸿沟正是工具接口：一份允许模型请求动作、由宿主程序执行的契约。2026 年的各类技术栈——OpenAI、Anthropic 和 Gemini 的函数调用（function calling）；MCP 的 `tools/call`；A2A 的任务部件（task parts）——本质上都是同一个四步循环的不同编码形式。本课将明确该循环的各个环节，并展示运行它所需的最小化机制。

**类型：** 学习
**语言：** Python（标准库，无需大语言模型）
**前置知识：** 第 11 阶段（大语言模型补全 API）
**时长：** 约 45 分钟

## 学习目标

- 解释为何仅能生成文本的大语言模型（Large Language Model, LLM）无法独立对现实世界执行操作。
- 绘制四步工具调用循环（描述 → 决策 → 执行 → 观察），并指明每个步骤的责任方。
- 将工具描述编写为三个部分：名称、JSON Schema 输入以及确定性执行器函数。
- 区分纯工具（pure tools）与副作用工具（side-effecting tools），并说明这种划分对安全性的重要性。

## 核心问题

大语言模型输出的是针对下一个词元的概率分布。这就是其全部的输出界面。如果你向聊天模型提问“现在班加罗尔的天气如何？”，它能生成一句看似合理的回答，但无法直接调用天气 API。这句话可能纯属巧合猜对了，也可能是三天前的过时信息。

弥合这一鸿沟正是工具接口的核心目的。宿主程序（你的智能体运行时、Claude Desktop、ChatGPT、Cursor 或自定义脚本）会向模型公布一个可调用的工具列表。当模型判定需要执行某项操作时，它会输出一个结构化载荷（payload），其中指明要调用的工具及其参数。宿主程序解析该载荷，实际执行工具，并将结果反馈给模型。此循环将持续进行，直到模型判定不再需要调用为止。

该契约的首个版本于 2023 年 6 月随 OpenAI 的 `functions` 参数发布。随后，Anthropic 在 Claude 2.1 中引入了 `tool_use` 代码块。几个月后，Gemini 也添加了 `functionDeclarations`。如今，所有提供商都暴露了相同的接口形态：输入为基于 JSON Schema 定义的工具列表，输出为包含 JSON 载荷的工具调用请求。模型上下文协议（Model Context Protocol, MCP，2024 年 11 月）对该契约进行了泛化，使得单一工具注册表即可服务于所有模型。A2A（Agent-to-Agent，2026 年 4 月 v1.0）则在此基础上叠加了相同的底层原语，用于智能体间的任务委派。

四步循环是贯穿所有这些实现的底层不变量。第 13 阶段的其他所有内容都是在此基础上的延伸与细化。

## 核心概念

### 第一步：描述（describe）

宿主（host）通过三个字段声明每个工具。

- **名称（Name）。** 一个稳定且机器可读的标识符。例如 `get_weather`，而不是 "weather thing"。
- **描述（Description）。** 一段自然语言简述。例如：“当用户询问特定城市的当前天气状况时使用。不用于历史数据。”
- **输入模式（Input schema）。** 一个描述工具参数的 JSON Schema 对象（draft 2020-12）。

模型会接收该列表。现代服务提供商（provider）会使用特定于提供商的模板，将这些声明序列化到系统提示词（system prompt）中，因此作为调用方，你只需处理结构化形式即可。

### 第二步：决策（decide）

结合用户消息和可用工具，模型会选择以下三种行为之一：

1. **直接回答**文本。不发起工具调用。
2. **调用一个或多个工具。** 输出结构化的调用对象。在 `parallel_tool_calls: true` 配置下（OpenAI 和 Gemini 默认开启，Anthropic 需手动启用），模型可以在单轮对话中输出多个调用。
3. **拒绝。** 严格模式下的结构化输出可以生成一个类型化的 `refusal` 块，而非调用请求。

工具调用载荷（payload）包含三个固定字段：调用 `id`、工具 `name` 以及 JSON `arguments` 对象。设置 `id` 是为了让宿主能够将后续返回的结果与特定的调用关联起来，这在并行调用返回顺序不一致时尤为重要。

### 第三步：执行（execute）

宿主接收到调用后，会根据声明的模式验证参数，并运行执行器（executor）。参数无效通常意味着模型产生了幻觉（hallucination）字段或使用了错误的类型——这在能力较弱的模型中是非常常见的故障模式。在生产环境中，宿主面对无效参数通常会采取以下三种措施之一：快速失败并将错误暴露给模型、使用受限解析器修复 JSON，或者将验证错误包含在提示词中让模型重试。

执行器本身只是普通代码，可以是 Python、TypeScript、Shell 命令或数据库查询。它会生成一个结果，通常是字符串，但也可以是任意 JSON 值或结构化内容块（如 MCP 中的文本、图像或资源引用）。该结果必须是可序列化的。

### 第四步：观察（observe）

宿主将工具结果追加到对话中（作为带有匹配 `id` 的 `tool` 角色消息），并重新调用模型。此时模型在上下文中拥有了工具输出，可以生成最终答案或请求更多调用。此过程会持续进行，直到模型停止发出调用，或宿主达到迭代次数的安全上限。

### 信任划分（The trust split）

出于安全考虑，工具可分为两类：

- **纯工具（Pure）。** 只读、确定性、无副作用。例如 `get_weather`、`search_docs`、`get_current_time`。可以安全地进行推测性调用。
- **后果型工具（Consequential）。** 会改变状态、产生费用或涉及用户数据。例如 `send_email`、`delete_file`、`execute_trade`。必须设置访问控制（gated）。

Meta 2026 年提出的智能体（agent）安全“双重规则（Rule of Two）”指出，单轮对话中最多只能结合以下三项中的两项：不可信输入、敏感数据、后果型操作。工具接口正是你执行该规则的地方——可以通过拒绝调用、要求用户确认或提升权限范围来实现。完整的安全章节请参阅 Phase 13 · 15，智能体级权限策略请参阅 Phase 14 · 09。

### 循环的载体（Where the loop lives）

| 上下文（Context） | 谁负责描述 | 谁负责决策 | 谁负责执行 |
|---------|---------------|-------------|--------------|
| 单轮函数调用（OpenAI/Anthropic/Gemini） | 应用开发者 | 大语言模型（LLM） | 应用开发者 |
| MCP | MCP 服务器 | 通过 MCP 客户端的 LLM | MCP 服务器 |
| A2A | Agent Card 发布者 | 调用方智能体 | 被调用方智能体 |
| Web 浏览器（函数调用智能体） | 浏览器扩展 / WebMCP | LLM | 浏览器运行时 |

无论在哪种场景下，都遵循相同的四个步骤。列名可能不同，但底层结构保持不变。

### 为什么不直接提示模型输出 JSON？

“要求模型以 JSON 格式回复”是函数调用出现之前的旧模式。在前沿模型上，其失败率约为 5% 到 15%，在较小模型上则更高。常见的故障模式包括缺少大括号、尾随逗号、幻觉字段以及类型错误。随后你通常需要执行 JSON 修复、重试或使用受限解码器。

原生函数调用（native function calling）更优，原因有三。首先，提供商会针对确切的调用结构对模型进行端到端训练，因此在严格模式下，有效 JSON 的生成率可攀升至 98% 到 99%。其次，调用载荷位于独立的协议槽位中，而非自由文本内——因此工具调用绝不会泄露到用户可见的回复中。第三，提供商通过受限解码（constrained decoding）强制实施模式合规性（如 OpenAI 的严格模式、Anthropic 的 `tool_use`、Gemini 的 `responseSchema`）。输出结果保证能通过验证。

Phase 13 · 02 将并排对比这三家提供商的 API。Phase 13 · 04 将深入探讨结构化输出。

### 熔断机制（Circuit breakers）

当模型停止发出调用或宿主达到最大轮次限制时，循环终止。生产环境中的宿主通常将此限制设置为 5 到 20 轮。超过此限制，几乎可以肯定模型陷入了无法退出的死循环。Claude Code 默认值为 20；OpenAI Assistants 为 10；Cursor 的智能体模式为 25。

替代方案——无限制循环——每隔半年就会以“智能体一夜之间消耗 400 美元 API 调用费用”的事后复盘形式出现。切勿在未设置上限的情况下发布产品。

Phase 14 · 12 深入探讨了错误恢复与自愈机制；Phase 17 涵盖了生产环境的速率限制。

## 使用它

`code/main.py` 在不依赖大语言模型（Large Language Model, LLM）的情况下运行四步循环。一个伪造的“决策器（decider）”函数通过对用户消息进行模式匹配来模拟模型行为；而执行器（executor）、模式验证器（schema validator）和观察步骤测试框架（observe-step harness）均为真实实现。运行该脚本可查看完整的请求/响应编排流程及可打印的中间状态，随后在后续课程中，你可以用任意真实的提供商（provider）替换该伪造的决策器。

关注重点：

- 工具注册表（tool registry）为每个工具保存以下字段：名称、描述、模式以及执行器引用。
- 该验证器仅使用标准库（stdlib）编写，是 JSON Schema 的一个最小子集（涵盖类型、必填项、枚举、最小/最大值）。第 13 阶段 · 04 将提供一个更完整的版本。
- 该循环将迭代次数上限设为五次。生产环境中的智能体（agent）正需要此类熔断器（circuit breaker）机制。

## 交付成果

本课程将生成 `outputs/skill-tool-interface-reviewer.md` 文件。给定一个工具定义草稿（名称 + 描述 + 模式 + 执行器概要），该技能（skill）会对其是否适合循环执行进行审查：名称是否具备机器可稳定识别（machine-stable）的特性、描述是否为完整的使用说明、模式是否正确遵循了 JSON Schema 2020-12 规范，以及纯函数（pure）与带副作用（consequential）的分类是否明确。

1. 在 `code/main.py` 中添加第四个工具 `get_stock_price(ticker)`。将其描述写为：“当用户通过股票代码查询当前股价时使用。不要用于历史价格或市场摘要。”运行测试框架（harness），并确认伪造的决策器会将提及股票代码的查询路由至新工具。

2. 破坏模式验证器。传入一个 `arguments` 对象缺少必填字段的调用，并确认宿主（host）在执行前将其拒绝。随后传入一个包含额外未知字段的调用。请做出决定：宿主应该拒绝还是忽略该字段？请从安全角度论证你的选择。

3. 将测试框架中的每个工具分类为纯函数（pure）或带副作用（consequential）。为需要该标记的注册表条目添加 `consequential: true` 标志，并修改循环逻辑，使其在选择带副作用的工具时打印一行“将向用户确认”。这就是每个生产环境宿主所需的确认门控（confirmation gate）的基本形态。

## 关键术语

| 术语 | 常见表述 | 实际技术含义 |
|------|----------------|------------------------|
| 工具 (Tool) | “模型可调用的组件” | 由名称 + JSON Schema 类型输入 + 执行函数组成的三元组 |
| 函数调用 (Function calling) | “原生工具使用” | 模型提供商级 API 支持，用于输出结构化的工具调用指令，而非自然语言文本 |
| 工具调用 (Tool call) | “模型发出的行动请求” | 模型输出的包含 `id`、`name`、`arguments` 字段的 JSON 负载 |
| 工具结果 (Tool result) | “工具返回的内容” | 执行器的输出，封装在具有匹配 `id` 的 `tool` 角色消息中 |
| 并行工具调用 (Parallel tool calls) | “一次性发起多个调用” | 在模型单次响应轮次中包含多个调用对象，彼此独立且可通过 `id` 排序 |
| 严格模式 (Strict mode) | “保证输出 JSON” | 约束解码技术，强制模型输出严格符合声明的 Schema 规范 |
| 纯工具 (Pure tool) | “只读工具” | 无副作用；可安全重复执行 |
| 具副作用工具 (Consequential tool) | “执行工具” | 会改变外部状态；需要设置审批关卡、审计或用户确认 |
| 四步循环 (Four-step loop) | “工具调用周期” | 描述 → 决策 → 执行 → 观察 |
| 宿主 (Host) | “智能体运行时” | 负责维护工具注册表、调用模型并运行执行器的程序 |
