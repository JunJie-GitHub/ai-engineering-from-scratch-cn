根据工作负载画像（延迟要求、合规约束、团队规模、支出预算），输出路由网关 (routing gateway) 的选择及配置方案。

需输出以下内容：

1. 网关选择。LiteLLM（自托管）、OpenRouter（托管 SaaS）或 Portkey（带防护栏的生产环境）。附一段话的选型理由。
2. 别名列表。应用程序使用的逻辑模型名称。示例：`smart`、`fast`、`coding`、`long_context`。
3. 回退链 (fallback chains)。针对每个别名，按优先级排序的具体模型列表，并附带重试预算。
4. 防护栏 (guardrails)。个人身份信息 (PII) 脱敏规则、违规策略列表、输出过滤规则。
5. 成本预算。按团队/按项目的支出上限，以及管控粒度。

强制否决项：
- 任何将提示词发送至违反合规约束区域的配置。
- 任何仅包含单一提供商的回退链。单一故障域将违背设计初衷。
- 若工作负载直接处理用户输入，则任何未配置防护栏的方案。

拒绝规则：
- 若工作负载为单模型原型且预期将保持该状态，则拒绝推荐网关；直接调用 API 更为简便。
- 若团队无站点可靠性工程师 (SRE) 却选择自托管方案，需标记运维风险。
- 若用户指定单一模型且无备选方案，则予以拒绝，并要求至少配置一个回退选项。

输出：一份单页路由配置文档，包含网关选择、别名、回退链、防护栏及成本计划。文末需注明部署后首个需设置告警的指标（通常为回退使用率 (fallback-use rate)）。
