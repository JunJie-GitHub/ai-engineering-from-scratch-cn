针对行为可能需要在调用中途（mid-call）获取用户输入的工具，设计提示收集（elicitation）的结构定义（Schema）与消息。

需产出以下内容：

1. 触发条件。明确说明应促使工具调用 `elicitation/create` 的具体输入或歧义情况。
2. 消息模板。宿主（host）向用户展示的一句话。要求表述通俗、具体且无专业术语。
3. 结构定义（Schema）。扁平化的 JSON Schema，包含带类型的属性以及 `enum` 列表（用于消除歧义）或 `boolean`（用于确认）。禁止嵌套。
4. 分支处理。将 `accept` / `decline` / `cancel` 映射到相应的工具行为。
5. 限流规则。限制单次工具调用中的提示收集次数；绝不在循环内部进行提示收集。

硬性拒绝条件：
- 任何嵌套对象的 Schema。Elicitation v1 必须是扁平结构。
- 任何用于填补大语言模型（LLM）本可通过自然语言询问的缺失参数的提示收集。
- 任何高频提示收集（单次工具调用超过一次）。

拒绝规则：
- 若工具为只读且风险较低，则拒绝进行提示收集，直接返回结果。
- 若工具具有破坏性且宿主支持 `destructiveHint` 注解，建议改用注解方式，并交由客户端原生处理确认流程。
- 若需求为 OAuth 登录，建议采用 URL 模式的提示收集，并标记 SEP-1036 漂移风险。

输出要求：一份单页设计文档，需包含触发条件、消息模板、结构定义、分支处理、限流规则，以及关于表单模式或 URL 模式何者更适用的说明。
