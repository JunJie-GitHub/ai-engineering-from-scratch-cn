给定企业级模型上下文协议 (Model Context Protocol, MCP) 规划（涵盖用户、后端及合规性约束），生成网关配置规范。

需生成以下内容：

1. 后端列表。每个后端需注明其注册源（官方 / Glama / 自定义）、规范名称（反向域名系统 (reverse-DNS) 格式）以及固定的描述哈希值。
2. 用户列表。每个用户需分配角色及允许使用的工具集。
3. 基于角色的访问控制 (Role-Based Access Control, RBAC) 矩阵。每行对应一个“用户 × 后端工具”组合，并标明允许/拒绝状态。
4. 速率限制 (Rate Limits)。包含单用户的突发与持续限制；针对高开销工具的按工具限制。
5. 审计计划。包含日志目标位置（文件、OpenTelemetry、安全信息和事件管理 (Security Information and Event Management, SIEM)）、保留周期以及需捕获的字段。

硬性拒绝条件：
- 未经管理员明确批准，任何未收录于官方注册源的后端。
- 任何允许所有用户使用所有工具的 RBAC 规则。这将导致权限爆炸 (Privilege Explosion)。
- 任何未采用不可变存储 (Immutable Storage) 的审计计划。这将导致合规性失败。

拒绝执行规则：
- 若开发人员数量超过 100 人但未定义任何角色，则拒绝执行引导 (bootstrap) 流程，并要求至少定义三个角色。
- 若规划中未指定 OAuth 2.1 身份提供商，则拒绝执行，并建议优先采用 Keycloak 或 Auth0。
- 若任何后端使用标准输入/输出 (stdio)，则拒绝通过 HTTP 网关进行代理；stdio 服务器应在每位开发者的本地环境中独立运行。

输出要求：一份单页配置文档，需包含后端列表、用户列表、RBAC 矩阵、速率限制及审计计划。文档末尾需附上团队应优先实施的一项策略规则。
